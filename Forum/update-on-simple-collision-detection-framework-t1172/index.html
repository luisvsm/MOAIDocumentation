/blog/moai-2012-roadmap.html

Re: Update on Simple Collision Detection Framework?

by jrabek » Tue Sep 04, 2012 4:48 pm

From this post it sounds like there was something there before.
collision-detection-in-moai-t167/

Would it be possible to see the old collision code as a branch just get an idea of how to get started implementing it?

I am looking at extending MOAI for detecting simple collisions (really sets of overlapping rectangles).

There seems to be two ways to do it.

Method #1: Mirroring the behavior of the box2d and chipmunk implementations.
1) Create a new MOAI class derived from MOAIAction, MOAICollsionSet, that has _insertProp, _removeProp, _setCollisionHandler. Since there is no physics simulation in the background, there is no need to create another object derived from MOAITransformBase as the parent as is the case with chipmunk and box2d.
2) When a prop is added to a MOAICollsionSet, the prop's parent becomes the MOAICollisionSet.
3) When MOAISim calls Update() on MOAIActionManager, at some point the MOAICollisionSet will be updated. After calling update on all its children it will check to see if any of it's child props bounding boxes overlap (via _getBounds).
4) For all pairs of overlapping props, the callback is made (set before with _setCollisionHandler).

Method #2: Create a new object outside of the action tree and node list that is called at the end of each sim step
1) Create a new MOAI class simply derived from MOAILuaObject, MOAICollsionSet, that has _insertProp, _removeProp, _setCollisionHandler.
2) MOAICollsionSet would basically be like a MOAIPartition with the key difference that once every sim step for all pairs of overlapping props a callback is made (set before with _setCollisionHandler).

Method #1 seems ugly just since there could be a collision during the MOAIActionManager Update() step which would be reported, but then during the MOAINodeManager Update() step where links are resolved the colliding props could be moved.

Method #2 seems ugly since the call to MOAICollisionSet to check all the props it contains for collisions would need to be jammed in after the MOAIActionManager Update() and MOAINodeManager Update() steps in MOAISim, or somehow jammed in at the end of the chain of node links to resolve in MOAINodeManager.

Any thoughts or guidance is appreciated.

Thanks!

I did a basic implementation of what I needed here:
https://github.com/jrabek/moai-dev/tree ... -collision

Others may find it useful. It uses an n^2 algorithm with the world bounds box for detecting collisions.
I have few objects and all my props are axis aligned so it works for me as is.

Future improvements:
* Add support for arbitrary polygons as collision shapes (MOAICollisionShapes looks like it may have been meant for this?)
* Implement an n log n algo for more efficient collision detection.

I also put up some sample code:

Re: Update on Simple Collision Detection Framework?

by jrabek » Tue Oct 09, 2012 6:05 am

I updated MOAICollisionSet to have static and dynamic props. Dynamic props are checked against all other props in the set. Static props are only checked against dynamic props (not other static props). This will speed up checking when there are a lot of not moving props on the screen.

Also, if a static prop has a grid (e.g. representing a map) then MOAICollisionSet will check the set of dynamic props for collisions with the cells in the grid with nonzero values.

For example, in a side scrolling shooter, the ships would be dynamic props, turrets would be static props, and the cave/world would be a static prop with a grid.
