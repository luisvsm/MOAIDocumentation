Re: [RELEASE] Html5/JS host

by halfnelson » Fri Sep 27, 2013 3:33 am

yeah it can be slow. Once you have run the bin\build-html script once this will have generated the makefiles under cmake\build.
Instead of running that script again (which deletes prior builds) you can change dir into cmake\build and just run make moaijs.

This will happily keep all prior build work and just compile what has changed. It will still take a while on the last bit since it is optimising and running the closure compiler. For faster builds when in dev you can go into the cmakelists.txt in the root of cmake and change the emscripten CXX and C flags to remove -O2 which should improve build times, you can also try adding --jcache to the list of flags and doing a full build again. Subsequent builds should be quicker.
(see faq entry here https://github.com/kripken/emscripten/wiki/FAQ)

Let me know if that helps and I might update the build task to have that by default. or add a flag which removes -O2 etc for rapid development

Re: [RELEASE] Html5/JS host

by halfnelson » Fri Oct 04, 2013 9:53 pm

Jdance, yeah, vc++ does macro expansion differently, under osx clang and linux gcc the ifdef is actually evaluated by the C compiler and never makes it into the shader source, only vc++ does the (imho) correct thing and passes the string through to opengl untouched. My work on a cli tool has led me to use mingw so I can distribute the compiler and I don't get this problem (sick of keeping 3 or more copies of visual studio on my box). It looks like the best course of action would be to do as you suggest since osx's opengl implementation complains very loudly about having a medium precision instead of MEDP. So that code needs to only show up on opengles implementations not full opengl ones.

I had a look at the injection thing (when i wrote the host), but didn't want to force a precision on peoples shaders and go to the trouble of determining if their shader already has one defined. It is really only needed for webgl and that is one of the only differences between webgl and opengles2.

Thanks for the tip, I might look at it again or feel free to have a go.
