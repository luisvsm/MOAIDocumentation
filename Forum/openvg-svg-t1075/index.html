Are there any plans to add some sort of OpenVG API / SVG support to Moai?

Evidently MonkVG has been built for iOS, Android, Mac OSX, *nix and Windows. It is an OpenVG like vector graphics API implementation that uses an OpenGL ES backend. MonkVG renders to a user supplied OpenGL context and is designed to play well with other OpenGL rendering (although GL state management is required in that case).

MonkSVG is canned SVG rendering solution that is built on top of MonkVG.

Re: OpenVG / SVG?

by javy » Fri Sep 07, 2012 9:32 am

I personally think that decompressing resources from SVG is the way to go. SVG isn't a magic bullet, but it might be a silver bullet as it can solve the big problem of exporting multiple images in the workflow. It also makes consumers happy as an app that would normally take 100 MB of space could take 20 instead.

I developed a decompression library for my iOS apps. I export an image to PDF, use ShrinkIt to reduce the file sizes, zip the files, then import them into the iOS app.

When the app is first run, a decompression screen is shown. The files are unzipped, then the PDF's are rendered to a category of UIView/UIImage. I then save the image as a PNG file and mark it as 'do not backup' for iCloud.

Obviously the big benefit of this is there is only one set of graphics based on the device (iPhone/iPad) and screen scale. In my tests I've saved about 80% of space with this approach.

In addition, it saves the need to export to various image sizes (or export one and reduce) as well as only needing one sprite sheet.

The drawback is two-fold, sprite sheets and iOS dependency.

Ideally there would be a single SVG version of this that works on both platforms (and future ones).

With TexturePacker, and the upcoming Spriter tool, I don't know how this system would be usable in a realistic workflow. There would need to be a tool that can re-create sprite sheets based on the coordinates exported by TexturePacker. There would also need to be a way to translate coordinates based on scale.

I'm very interested in this endeavor, unfortunately my SVG coding skills are highly lacking.

Hi, maybe this might interest you:
I'll be posting a lua-based SVG importer very soon, which will have following features included:
- base SVG objects support: rect, circle, polygon, polyline, bezier path, text, multiline text and image
- solid color fill, gradient fill or texture fill
- lines can have width as will be created as stripe mesh object, with custom UV tiling (so you can have texture repeat along line)
- texture packer support with animated sprites
- physics: objects can be assigned physics properties or declared to be joints
- custom behaviors

svg importer will be part of the very simple entity-component system, so custom behaviors can be easily coded and properties can be set through Adobe Illustrator's SVG Interactivity panel, like:
name:bird follow:@path4,100
where object named "bird" would follow a path named path4 with speed of 100 pixels per second (for example)...

I have been long overdue with this importer because I'm busy with client projects, but hopefully it will
come together very soon...

Re: OpenVG / SVG?

by nosheet » Thu Sep 13, 2012 8:24 am

I'm now humbled with high expectations, hehe Actually I have been crazy busy for months and was able to do this only some 1hour a week, so it's coming of slowly. Also, the way I am focusing this, is to use it as a level editor and visual game logic editor, and haven't even thought about possibility to use SVG to optimize assets (probably because I am used to create all my assets in Photoshop, in super high-res vs. working with vectors).

I am writing this because some people might wait for this to come out only to realize it was not what they needed.
So here's a few hints on what's going on here:

At first, when I mention Texture Packer support / spritehsheet support, I am not referring to loading SVG and exporting various image sizes. Instead, the typical use case scenario would be like this:
Imagine we want to lay out Space Invaders game in Illustrator. To simplify, let's say we have a player.png and just one type of alien - alien.png.
We import these two images in Illustrator, and copy to make lots of aliens, and then save the SVG.

Now, having separate images/textures for each sprite obviously sucks, and for that reason we want to use TexturePacker (or any other similar tool) to pack both player.png and alien.png into one texture.
Assuming we created invaders.lua and invaders.png spritesheet files, we can add:
Code: Select all
  1. sheet:invaders


in the name of the Illustrator group that contains alien.png and player.png, and so SVG parser will know that instead of looking for image files it should load invaders spritesheet (if not already loaded) and look for frames called alien.png and player.png.
Additionally, parser will check the view scale and if needed, check if there is a high-res version of the spritesheet@2x and load that instead.
Also, if we already created images for animation frames and packed them into our invaders.lua/png spritesheet, we can define named animations inside of Illustrator and call them up later in our MOAI Lua code (more about this in detail once it's out)

Again, in my workflow I still export spritesheets of different resolutions manually, because I am used to creating raster images.



More importantly (for me) and probably more confusingly (for the rest of the world), focus of my SVG importer is to read .svg file and convert it to list of data entities ready to be fed in Entity Component System (ECSess) that I wrote.
Therefore, important part of my "workflow" is to use SVG Interactivity panel in Illustrator to assign variables to objects, the ones I will access later in my Lua code.

So for example, I can write down following in SVG Interactivity properties of player.png object:
Code: Select all
  1. bhv:tapControl speed:5


... and entity component parser will look for tapControl.lua module that contains logic that needs to be applied to this entity, and the entity will be initialized with a variable speed=5.
The idea is to have a library of commonly used components/behaviors ( various types of player controls, steering behaviors, collisions or even GUI behaviors such as buttons, etc...) that can be easily reused for various projects (or more likely, quick prototyping), each behavior in its own .lua file, being served the variables and values I write down in SVG.

In line with this, SVG importer doesn't actually draw objects, it merely reinterprets SVG content into format ready for use by ECSess system, and assignes "display" behavior to all of them by default.
Display.lua "component" is the code that actually draws everything on screen, but it's just a one of many behaviors an entity can have.
So for example if we write down in SVG Interactivity panel when having our player.png selected:
Code: Select all
  1. id:spaceship bhv:ph,tapControl den:1.5 bnc:0.3 speed:5 shape:@frog



This would mean that ECSess will look for ph.lua, which I wrote to handle physics, so it will automatically create rigid body with density of 1.5, bounce 0.3 and then it will search for polygon from the same svg file that we named "frog" and use it a a fixture for the rigid body... bunch of other properties ares supported, like collision masks etc...
Also, since we added id:spaceship, we can access this entity from anywhere in our code using:
Code: Select all
  1. ECS.GetEntityByName( "spaceship" )


, although the idea behind ECS system is that most of our game logic is modular in self-contained .lua files, so that in our main game logic layer we may even never need to access this (or any other) entity directly.

I am well aware this probably sounds very confusing, but it's actually quite fun to use, and it should be crystal clear with a few examples I will add.
I might be getting a new client project soon, where I would need to create bunch of mini games in a short period of time, and I am planning to use this SVG + ECS workflow, so this is probably when this will finally get more of my attention instead of being an hour-a-week pet project of mine.
