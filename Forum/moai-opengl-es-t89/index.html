Moai + OpenGL ES

by patrick » Tue Aug 09, 2011 11:37 am

1.

Would be interested in talking more about that. I can link in an off the shelf OpenGL/Lua binding without a lot of trouble. Moai's implementation is a retained mode for OpenGL that does some optimization via state change caching and tries to ease compatibility issues between fixed function and programmable for games that use both.

Here are some thoughts re how to proceed. One path would be to expose the OpenGL API and then create some extensible objects that let you call OpenGL directly from inside the scene render. So you'd have a prop, deck and layer object with a callback to let you do native rendering. That way you could still use Moai's animation and dependency graph to organize your scene, but render however you want to.

The other approach would be to add more retained mode objects that can efficiently render vector style graphics, advanced effects or whatever you want. If we take this approach I can make sure those objects play well with our optimization path.

The approaches aren't mutually exclusive and I'd eventually like to support both. Let me know what's important to you and I'll keep it in mind as we move forward.



2.

If they are Lua modules, absolutely. I'll eventually be doing a DLL configuration of Moai for desktop so you can load native modules as well, but given our focus on mobile in general and iPhone in particular it hasn't been our top priority.



3.

That would be fantastic. Most of the team here has been using Notepad++ and print (), but a good builder tool/IDE would be appreciated by a lot of people. We don't have the bandwidth to tackle it internally right now. There are a few tools projects out there - a better desktop host from Bach Le and an IDE from James Rhodes - you might want to ping them first.

...

Glad you are digging Moai.

Feedback is hugely appreciated. It's still young technology, so hearing back from people using it helps us a lot. And very cool to get the perspective of someone at a top shelf studio like Pandemic. Thanks for reaching out!

Moai + OpenGL ES

by patrick » Tue Aug 09, 2011 7:05 pm

Huzzah!

1. That's right. It's a good way to gloss over some of the cross platform quirks. Also, it lets us do optimization tricks like sorting on draw state, batching draw operations and viewport culling. Will be even more relevant down the road when we need to support things like skinning and occlusion.

We're pushing out a new release tomorrow or the next day that will support programmable pipeline. So my comments below apply to that release, not the current version...

You can set up a custom vertex format using MOAIVertexFormat. This object does double duty for fixed function and programmable pipeline. If you don't care about fixed function at all, you can just describe your vertex attributes directly. If you want to be backward compatible with fixed function, you use the named vertex component declarations. MOAIMesh is a special kind of deck that binds a vertex buffer to a texture. You then attach the mesh to a prop to place in your scene and render it. If you want to draw using your own shader, you can attach the shader to either the deck or the prop. When you're setting up your uniform bindings to the shader you can link them to custom dep node Attributes (on the MOAIShader object), MOAITransform objects, or set them up to be populated with matrices from Moai's transformation pipeline (so you can get your model, view, projection transforms directly).

The decks right now don't support multitexturing; that's something I need to add. As you work more with it, would be interested to hear about what effects you need to create so I can revise/extend the retained mode to handle them. Also, check out the next release and see what you think of the new programmable pipeline stuff. You should be able to do a *lot* more without giving up retained mode.

As for the Lua/GLES binding you wrote, heck yeah. If you put it up on github under a license we can use (MIT or something like that) I can grab it and look at linking it in as an extension. Alternatively, if you fill out our contributor agreement you can send it to me directly. For us to support it, should be as simple as adding some rendering callbacks you can service from Lua.

2. Haven't. Odd that they would disable modules. I'll have to look more at dll/.so configuration - though may not get to it for a while given the mobile focus at the moment.

3. Very cool. A one click builder would be a big help to a lot of devs. I know first time Xcode/Eclipse/Visual Studio users have struggled coming to Moai for lack of a dedicated IDE/builder. The other thing that would help those devs would be a high level wrapper written in Lua. Mainly a scene manager, but also a friendlier game object that wraps up props, decks, textures and physics objects and hides things like UV coordinates, etc.

...

Yeah, let us know how we can help you move that project along. Sounds like you have a built-in audience for it!
