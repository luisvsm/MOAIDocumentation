Alpha and fade to color

by patrick » Wed May 04, 2011 5:17 pm

You bet. The way we do it now is to attach a shader to the prop:



shader = MOAISimpleShader.new ()

prop:setShader ( shader )

shader:setColor ( 1, 1, 1, 1 ) -- R, G, B, A

shader:seekColor ( 0, 0, 0, 1, 1.5 ) -- fade to black



Moai uses premultiplied alpha, so to fade to nothing:

shader:seekColor ( 0, 0, 0, 0, 1.5 )



And to glow:

shader:seekColor ( 1, 1, 1, 0, 1.5 )



MOAIShader is just an alias for the fixed function pipeline. It's a placeholder for the programmable shaders coming with OpenGL ES 2.0 support.
I'm still thinking about the decision to move color info into the shader node. I may refactor this before release. But for the near future let me know if this does the trick for you.

Alpha and fade to color

by patrick » Fri Oct 28, 2011 3:57 am

Sure. In this case, Moai is expect premultiplied alpha, so 'BLEND_NORMAL' is set up for that. The texture loader also defaults to premultiply alpha at load time; you'll want to turn that off and do it in a resource build step (or let Xcode do it).

If you don't want premultiplied alpha, you should be able to set the blend mode directly (without using the preset).

by patrick » Tue Dec 06, 2011 5:04 pm

So it works for me fine under the glut host. It *should* work fine on any host. There's shouldn't be any problem with multiple actions animating the color - multiple animations just get run at the same time by the action manager. The first seekColor should detect the delay of 0 and just do a setColor under the hood. The second seekColor should spawn an action and run business as usual.

Which host/platform are you running under, and have you verified that the logo shows up OK if you don't do any fading?
