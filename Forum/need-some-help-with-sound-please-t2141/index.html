Re: Need Some Help With Sound, Please

by halfnelson » Tue Nov 05, 2013 9:41 pm

When I was writing the untz support for the web, I was trying to work out what the native modules were doing. It seems that to reduce stuttering they introduce a small amount of buffering when you go to play the sound. I think reducing the size of this buffer is key (this will probably differ on different native implementations). The reason upsampling works is that the buffer size is probably worked out as a number of samples, and the increased sample rate reduces the time that the buffer represents. I would have a look at any of the the Buffer allocation/constants in the native implementations and try reducing them. For short sounds the stuttering probably wont be a problem, but for music loops it might cause issues.

I could be way off, but I was aware of the lag when looking through the source so kept that in the back of my mind as I read. The documentation for rtaudio which the windows host uses also mentions the buffer and latency too.

The other thing to try is to use SDL in windows and see if the SDLMixer solves the problem ( I am sure that there have been rhythm games written in SDL

Re: Need Some Help With Sound, Please

by halfnelson » Thu Nov 07, 2013 1:37 pm

had a closer look at the source for native ios and compared to other projects using it for low latency sound and found that the call to "kAudioSessionProperty_PreferredHardwareIOBufferDuration" as defined here:

https://developer.apple.com/library/ios/documentation/Audio/Conceptual/AudioSessionProgrammingGuide/UsingAudioSessionProperties/UsingAudioSessionProperties.html#//apple_ref/doc/uid/TP40007875-CH13-SW1

is set to frames/samplerate
Code: Select all
  1. float aBufferLength = (numFrames+1)/(float)sampleRate; // In seconds



defaults are 8194/44100 = 0.185 seconds!!!

so possible fix is to just change the line to
Code: Select all
  1. float aBufferLength = 0.02; // In seconds



from ios docs
The default audio I/O buffer duration (about 0.02 seconds for 44.1 kHz audio) is appropriate for most applications. It provides enough responsiveness for any sort of user interaction, including fast-action games. Do not set a lower I/O duration unless your application needs it


EDIT: looks like untz for windows is almost identical problem, it buffers 8192 frames by default (so same lag as ios). The rt audio library it uses allows you to pass 0 for number of frames, and it will determine the lowest latency it can get and use that. That might fix windows sound lag too.

@nosheet: I'm glad it worked, I didn't have time to try it out. Looking forward to trying your game.

@ezra:
it looks like this (on line 89 UntzSystem.cpp):
Code: Select all
  1.  

  2. wsd->audioIO.openStream( &outParams, NULL, RTAUDIO_FLOAT32, sampleRate, &numFrames, &RtInOut, (void *)&mpData->mMixer, &streamOptions );

  3.  



I think numFrames might be used for other calculations, so instead of setting numFrames to 0, pass a reference to a uint32 that contains 0. This will be updated with the actual size of the buffer in frames (after the call completes)

From RtAudio.h
\param *bufferFrames A pointer to a value indicating the desired
internal buffer size in sample frames. The actual value
used by the device is returned via the same pointer. A
value of zero can be specified, in which case the lowest
allowable value is determined.
