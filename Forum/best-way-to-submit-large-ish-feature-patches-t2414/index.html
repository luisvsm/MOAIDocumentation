best way to submit large(ish) feature patches?

by lordmortis » Mon Dec 16, 2013 1:14 am

We've been working on a game that uses Moai (Freedom Fall - http://www.freedomfall.com) that's available on Ouya, Android, iOS, Mac OS X and Windows (Linux forthcoming) and we've made a few changes to Moai that make it more useful across these platforms.

These changes include:
  • SDL 2.0-based host for desktop support (Mac OS X, Windows (and Linux)) (which we call "Sledge")
  • Controller support on Android, Ouya and desktops (Via SDL2) (iOS support forthcoming)
  • OUYA Purchase API Support
  • OUYA Deployment support (OUYA dev key specification,
  • Asynchronous JPG/PNG image loading/decoding.
  • Actual Asynchronous web requests (CURL blocks on the desktop platforms)
  • JPG/PNG "texture compression" system (Loads RGB data from JPGs and Alpha data from PNGS resulting in smaller binaries)
  • Hooked up many MOAIEnvironment variables on iOS, Android, Windows and Mac OS X allowing access to number of CPUs, build number of game, version of game, cache directory, documents directory etc.
  • ZeroCopy MOAIImage -> MOAITexture support (when creating a MOAITexture from a MOAIImage it no longer copies the buffer, it just reassigns it to the MOAIImage belonging to the MOAITexture)
  • Opus Audio Support (See: http://www.opus-codec.org)
  • VS2012 project
  • Husky System for cross-platform achievements (We have a Steam Husky available)

We initially branched off moai-dev master branch in feburary of last year, but recently merged all the changes while preparing for our steam release (so we may have merged the Garbage Collector error, although we've not yet encountered it)

At this stage post-mega merge, we haven't tested the iOS and Android runners but we will be releasing updates to Freedom Fall for Ouya, iOS and Android with controller support and achievement support (at least on iOS) in February so we'll be fixing it up by then.

We're have working Steam Achievements, Leaderboards and Cloud Saves via the Husky external library system (due to Valve NDA).

We're working in this Github repository here: https://github.com/StirfireStudios/moai-dev
We have hosts as detailed in this post: /forums/best-way-to-submit-large-ish-feature-patches-t2414

If you have any questions about what the above changes entail, please don't hesitate to ask (or browse the source)

We'd like to contribute this back to upstream once we do our updates for iOS and Android post-steam release.

I also realise there's big changes coming in Moai-land due to the merging of several other forks but I'd like to start this discussion now, so when we are ready to give some time to creating patches and merging things, there's a plan

Re: best way to submit large(ish) feature patches?

by lordmortis » Tue Dec 31, 2013 2:00 am

By the request of some people here, I've uploaded some pre-built hosts for Mac OS X and Win32

Warning: These hosts are not fully tested and almost certainly contain bugs. Plus the Win32 host is currently NOT winXP compatible (as we've been working on a VS2012 project to build moai, but haven't worked out all the kinks yet)

I'll keep posting as we update hosts and the moai-hosts branch will get updated when we finally produce release binaries for win32.

I know that our branch contains a Ogg Vorbis playback bug in win32 and there may be other issues lurking.

Directions:

Win32 https://files.app.net/dh2g20jW.zip
Commit: fbd79709914555d863afa714231fa702d6b685b7
  • extract the zip file to wherever you run moai from.
Mac OS X https://files.app.net/dh2hiIcT.zip
Commit: 3d1394cfb5a313b874474ea096b54fc0213861cc
  • extract the zip file to wherever you run moai from.
  • if you don't have the SDL framework installed, stick it in your ~/Library/Frameworks folder
Both:
  • put your main.lua (or compiled main.lb) file either in the same dir as the moai-sdl directory, or in a Resources/ subdirectory
  • alternately supply the main lua file on the command line
