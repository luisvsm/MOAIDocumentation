Re: Regular GUI app support

by thebullno1 » Wed Jan 23, 2013 3:24 pm

mcandre wrote:Can you write nongame apps for Android using Lua/MOAI?

You can but the out of the box features are limited. There are no native GUI support. You can create your own GUI library or use some community ones like moai-gui. Features like Camera are not built in (but it's open source so you can add).

Will MOAI support Windows RT so that Lua coders can write Surface and Windows Phone apps?

Not sure about this. Moai code is quite clean and all gl calls are isolated to 1-2 files. I don't think it's difficult to port the renderer to DirectX. You can put a GL emulation library on top but performance might not be that great. Most modern GPU works on the same principle: give it a rendering function (a fragment and a vertex shader), give it some input (vertex/index buffer, texture buffer), it will render the thing for you.

A large part of Moai's fixed pipeline emulation will need to be rewritten in HLSL though.

by seebs » Fri Feb 01, 2013 4:58 am

I'd point out that a library of simulated GUI objects does not solve some key issues.

For instance, on iOS, VoiceOver (the accessibility stuff for people who can't see well enough to use the standard UI elements reliably) works with native elements. So if you have an actual iOS button, the user can use it. If you have anything else that is merely a screen area which responds when tapped, they can't.

Writing a consistent framework that would work for multiple targets and let you describe a GUI is, however, fairly hard.

Re: Regular GUI app support

by ibisum » Fri Feb 01, 2013 4:06 pm

One thing you should keep in mind is that MOAI, in the iOS host, is running in a GLView. There is nothing stopping you from surrounding that GLView with the native components you need, and then using ViewControllers in the iOS host to move things around as your app needs. So if you want a native button to be part of your app, put it in the iOS host (programmatically, not with .nib's) and then wire up some code to cross the LuaVM->Native boundary to get events on that button - its as simple as "implementing it natively first, then wiring up a control method to the LuaVM", which is .. just one more step beyond doing it natively.

A close look at the iOS host code will reveal that in fact there already are a few native controls in your typical MOAI app - one for the Keyboard i/o, for example. This same device can be used to support other native GUI controls if you need them - you can certainly use ViewControllers, standardly, to add native GUI elements to a typical MOAI app. You just have to know how to do it natively, anyway ..

Well, this is the "roadmap" forum, no?

Seems to me that "wrappers for native GUI objects" are the sort of thing that would increase the value of the SDK to a lot of people. I've done Objective-C apps, and I've done apps in Corona, and I've done UI code in a bunch of other frameworks, and... I guess it's a philosophical question: What's the goal of the SDK? Purely to provide for Lua access to OpenGL, or to provide for full apps written in Lua where possible? Re: Regular GUI app support

by ibisum » Sat Feb 02, 2013 3:13 am

Well we're chasing tails here, but if you want a nice set of GUI controls, moaigui can provide you this - all the basics are there, from sliders to buttons to text fields, checkboxes, radio buttons, and so on. There's even a linegraph control. Hanappe, also, has a lot of the primitives - and while neither of these frameworks uses native code for the GUI presentation layer (its all just pure MOAI), they can be used to make perfectly acceptable GUI's, given the right motivation on the part of the developer. If you want more than what these MOAI-based frameworks deliver, then of course: you have to add those features to the MOAI host yourself, or implement the new stuff in raw, plain MOAI ..

And yes, the door is wide open for someone to add GUI widgets to the main Application view in either the Android or iOS hosts - and some have already done this for their own local projects, myself included. When/how this all gets promoted back upstream to main-dev, is an open question however .. but the fact is that if you want to add more advanced GUI features, and if you know how to do it natively anyway, then you can most certainly add the feature in a generic way which will suit the cross-platform requirement. Just look at the existing code for things like MOAIDialog*, and so on .. the plumbing and infrastructure is already in place, for the most part you just have to decide on a sensible way to expose it within the context of the Lua VM, and then get on with it ..

We've already determined in the forum that Zipline have other priorities that relate to the survival of their business, and one of those priorities is doing extensions to the MOAI host for paying customers - if you really need something for your project, I'm positive Zipline are available for the right price. Or, you could try shopping around here too - there are by now enough power MOAI users that you can name your price, for the most part, and probably find someone willing enough to do the plumbing work ..

Re: Regular GUI app support

by seebs » Sat Feb 02, 2013 4:48 am

I am pretty sure that the available budget for my pro-bono work that has brought in $35 of donations so far is not going to cover a major new development.

I think the key here is: "acceptable" is contextual. Acceptance is related to specific criteria, which vary widely.

The criterion I am most concerned with, by far, for GUI items is accessibility, and that means that the operating system knows what text is in a text field, and knows that a given thing is a button and what the label of that button says. That's something that can only be done with native widgets -- meaning, the widget API used by the host OS.

No amount of motivation or effort can ever turn a thing which wasn't created using the host's button API into a thing the host recognizes as being a button.

And yes, I can in theory go ahead and write wrappers in each of the target systems for buttons. I may even find that doing so is the best way to get what I want. But I want to point out: There are real technical reasons to care whether something is a "native widget" or not. That is one of the few things which you really cannot implement in terms of other primitives.

... Which actually leads to a sort of interesting thought.

One of the tools out there for iOS/Lua development is a very clever little hack in which someone uses introspection to dynamically bind to... everything. You can call any arbitrary objective-C thing from Lua, because you can use introspection to get the bindings in place. (I may have details wrong; LuaCocao reference here; also has some references to other similar projects.)

It might make sense, fitting with Moai's love of giving you primitives from which you can build things, to toss together a fairly simple thing based on something like that, and make it available as a lower-level tool; this would provide a mechanism for getting at "and all the functionality no one has done more specifically yet".
