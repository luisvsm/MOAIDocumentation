When I load a new level, sometimes I get an assert crash:

Code: Select all
  1. 3rdparty/box2d-2.2.1/Box2D/Collision/b2DynamicTree.h:159: const b2AABB& b2DynamicTree::GetFatAABB(int32) const: Assertion `0 <= proxyId && proxyId < m_nodeCapacity' failed.



Code: Select all
  1. inline const b2AABB& b2DynamicTree::GetFatAABB(int32 proxyId) const

  2. {

  3.         b2Assert(0 <= proxyId && proxyId < m_nodeCapacity);

  5. }



This never happens the first time I build a level, but, only randomly on next level build, so it could crash after playing 5 levels or 2. It's almost like there is a hanging reference somewhere, but even if that was true, if I've stopped the physics world and not updating it, and built a new world, how could the old one be updating? Maybe there is a prop still using an old body as a parent somewhere... but I'm clearing and rebuilding all layers...

Woudl there be negative consequences to disabling this assert?

Any thoughts on how to track this down? How to find a hanging reference when you have no idea where it might be?

Re: Box2D GetFatAABB Crash: Advice Needed

by stun » Mon Nov 17, 2014 7:44 pm

Random suggestion, but I'd split the assert into two asserts as the first step. Is the proxyID 0 or lower, or is it above your nodeCapacity for Box2D? It certainly sounds like something isn't getting removed from the Box2D world.

And the assert is in there for a reason...removing it might not be the best idea without finding the root cause. Removing it means you'll be potentially referencing memory OUTSIDE the m_nodes array...which if it doesn't crash, will return you a garbage AABB structure...which is most certainly not what you want.

My other suggestion is to cram good old 'printf' type statements in there and capturing the output and searching for your offending proxyID. Printf type statements are crude I admit, but they almost always will produce useful information in my experience.
