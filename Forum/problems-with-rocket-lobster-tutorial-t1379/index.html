Re: Problems With Rocket Lobster Tutorial

by paw9000 » Thu Dec 06, 2012 8:28 am

Figured it out.

I was checking out the "moai-sdk\samples\input\input-picking\main.lua" file which happens to run just fine on both iOS and Windows and noticed they check to see if MOAIInputMgr.device.pointer is non-null before calling the setCallback. If null then they use the MOAIInputMgr.device.touch:setCallback instead. I plugged similar logic into the lobster main.lua and now it runs on iOS as well. Hooray.

by stun » Thu Dec 06, 2012 7:40 pm

Ibisum, I'm running MOAI using the default GLUTHost build with Visual Studio 2010 in the 'debug' configuration. The MOAI VS2010 solution that exists in the repo generally just builds out of the box (unless you're building a more exotic target like moai-fmod-ex). Here's what I'm doing step-by-step. Apologies if some of these steps seem rather obvious, but maybe this will be useful to others in the future.

Debugging the MOAI GLUTHost in Visual Studio 2010/2008:

1. Build 'moai' or 'moai-untz' in the 'debug' configuration and just ignore any errors you get on the other projects. The configuration you're building is typically in the toolbar at the top of the IDE. You want 'Debug' and 'Win32'.

2. Once you have a debug version of 'moai.exe', using the IDE right click on the 'moai' project in the solution. On the context menu select 'Set as Startup Project'.

3. Right click the project again for the context menu and select 'Properties' on the context menu and go to the 'Debugging' entry. Set the 'Command Arguments' to be the name of your MOAI lua script: 'main.lua' typically. Set the 'Working Directory' to be to directory on your system where 'main.lua' and all the related resources for your game live as well.

4a. Set some breakpoints in the MOAI classes you're interested in debugging and then hit 'F5' to start everything up.

4b. Alternately it's quite interesting to at least once just hit 'F10' and step through the whole host startup process from scratch.

I've found interactively debugging the host is absolutely invaluable for extending it, and/or understanding what is actually going on internally in MOAI. If you're building from the source you can of course do exactly the same thing using gdb under Linux, or the Windows equivalent windbg. I haven't used gdb in years, but windbg is a tool that is incredibly powerful, but also super esoteric (and basically command line like gdb). Visual Studio is a little more friendly to use for debugging, but windbg is essential for port-mortem debugging. I haven't released a game yet, so I haven't had to do any post mortem debugging on MOAI crashes under Win32 yet thankfully .
