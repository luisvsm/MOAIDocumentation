Xbox Controller support in PC Host added

by stun » Sun Dec 16, 2012 10:41 pm

Hello, for anybody that cares I've added an first cut of Xbox controller support to the PC host. It's available on my personal fork here: https://github.com/c0d3monk33/moai-dev

It consists mainly of two files (and some tweaks to the VS2010 solution to reference the Microsoft xinput.dll): MOAIXboxController.cpp and MOAIXboxController.h. It's not an ideal solution as MOAIXboxController is derived from MOAIAction (meaning you'll have to start() it). Ideally I'd like to create a better AKU/sensor based implementation, but I'm still trying to determine how to do so. It works for me now so I'm posting it here. The code supports:

- 1-4 Xbox controllers.
- All the buttons (with the exception of the large 'XBOX' button which xinput doesn't expose), sticks and triggers on the controller.
- Callbacks and polling methods for examining controller state.
- Simple vibration support.
- Doxygen commenting on the MOAI exposed functions.

Hopefully it should be fairly obvious how to use the class, but I'll post a simple test app at a later date.

by thebullno1 » Thu Dec 20, 2012 10:23 am

stun wrote:No worries. I created it mainly out of curiosity more than anything. Unfortunately it doesn't fit the current AKU model of 'sensors', because I haven't spent the time to figure out how to represent multiple, distinct groupings of sensors for each controller - to be honest I'm not entirely sure AKU supports this as it stands from the cursory look I've taken.


That's what deviceId is for.

AKUReserveInputDevices(n) creates n empty devices. Each device is a controller. Usually, we only use a phone so n is 1.
AKUReserveInputDeviceSensors(deviceId, n) creates n sensors in device with id: deviceId (start with 0). For a controller, each button and axis are considered a sensor. Digital buttons can be easily mapped to AKUSetInputDeviceButton. The analogs are tricky because there's no direct equivalent. If I am to do it, I will use AKUSetInputDeviceWheel for triggers (since they are 1D). and AKUSetInputDevicePointer for the thumb sticks since they are 2D.

Hot plugging is another problem, AKU does not support dynamically adding or removing devices. You can preallocate around 4-5 devices and set them as active/inactive on the fly.

The controllers work as they stand now, but are completely divorced from the standard MOAI AKU code which I feel is a little poor. Although of course they're only ever going to work on the Windows PC host with the required Microsoft xinput.dll, but I would like to do it properly at some point!

I did notice MOAI does have a 'MOAIJoystickSensor' which represents the stick direction as a vector.

Here's a quick script to test the existing controller code too:

Code: Select all
  1. -- Test an Xbox controller

  2. controller = MOAIXboxController.new()

  3.  

  4. function xboxButton(controllerId, button, down)

  5.   if down then

  6.     print("controller " .. controllerId .. ": button: " .. button .. " DOWN")

  7.   else

  8.     print("controller " .. controllerId .. ": button: " .. button .. " UP")

  9.   end

  10. end

  11.  

  12. function xboxTrigger(controllerId, trigger, value)

  13.   print("controller " .. controllerId .. ": trigger: " .. trigger .. " value: " .. value)

  14.   controller:setRumble(0, value)

  15. end

  16.  

  17. function xboxStick(controllerId, stick, normalX, normalY, mag)

  18.   print("controller " .. controllerId .. ": stick: " .. stick .. " mag: " .. mag .. " dir: " .. normalX .. ", " .. normalY)

  19. end

  20.  

  21. controller:setId(1)

  22. controller:setButtonCallback(xboxButton)

  23. controller:setTriggerCallback(xboxTrigger)

  24. controller:setJoystickCallback(xboxStick)

  25. controller:start()
