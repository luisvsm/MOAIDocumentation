Re: jerky / hitchy / jittery scrolling under Windows, iOS

by patrick » Tue Feb 21, 2012 8:11 pm

Also should mention, on iPhone and iPad there was an occasional hitch, but when running release builds (off the test Matthew supplied) the hitch was *barely* noticeable. Debug builds were considerably worse.

@mikegriffin The loop flags should be respected - check out MOAISim.cpp to see the logic. What the sim does *not* do is average performance and try to lock the framerate to a lower frequency. The sim and the hosts right now just try to run as fast as possible. As Adam said, on some devices we're seeing a lot of thrash between 60hz and 30hz, even with no rendering or sim update. So unfortunately, the best fix may be to sample the update speed, average it and lock at the low end.

All of this stuff is like... decide how you want your punishment. Locking at a low frame rate will make your updates consistent but your animation choppy. Boosting with a variable frame rate will cause skipping and may blow up your physics. Spinning with a fixed frame rate will cause hitching unless the step is very small - which is going to be processor intensive. And none of this has much to do with how fast the sim or render run - on some devices we have these problems even if our update and draw run well under 60hz.

We're still looking into this and are hoping to have some breakthrough. The tough part is, even though the symptom is the same, the cause is highly platform dependent and could be the result of different things on each host. Even the GL implementation factors in, which is why glFinish matters on some devices but not on others.

Re: jerky / hitchy / jittery scrolling under Windows, iOS

by patrick » Tue Feb 21, 2012 10:53 pm

That's interesting. I'd have to look at the code again. There is actually a 'timer error' threshold in MOAISim that can be used to account for small variations like that. It defaults to 0 and is still experimental, but the intent was to effectively snap the intervals to the target frame rate if they were just a little behind or ahead.

Another note: Adam pointer out that the flags implementation of MOAISim is inconsistent with bitflags handling in other parts of the SDK. MOAISim.setFlags () does a bitwise OR with the current flags instead of overwriting them. So you have to use MOAISim.clearFlags () first.

Re: jerky / hitchy / jittery scrolling under Windows, iOS

by patrick » Wed Feb 22, 2012 12:09 am

What Adam's done there is to use the loop flags to enable a fully variable frame rate. When the boost threshold is 0, every step triggers a boost which will pass the delta since the last update to the sim.

MOAI's built-in animation system can handle a variable frame rate. Just keep in mind that you may see physics stability issues with this method and you'll also have to be careful when writing stepwise movement code in Lua.

We are still working on this internally and will be testing a variety of options.

This is my understanding -

The main loop or heart of the system is in GlutHost.cpp

Code: Select all
  1.        

  2. static void _onTimer ( int millisec ) {

  3.         UNUSED ( millisec );

  4.  

  5.         int timerInterval = ( int )( AKUGetSimStep () * 1000.0 );

  6.         glutTimerFunc ( timerInterval, _onTimer, timerInterval );

  7.        

  8.         AKUUpdate ();

  9.        

  10.         #ifdef AKUGLUT_USE_FMOD

  11.                 AKUFmodUpdate ();

  12.         #endif

  13.        

  14.         glutPostRedisplay ();

  15. }

  16.  

  17. static void _onPaint () {

  18.        

  19.         AKURender ();

  20.         glutSwapBuffers ();

  21. }

  22.  



The flows are:
akuupdate -> sim.update -> sim.step (which is where all the sim flags are for variable frame rate - these work by looping back into sim.step to try and catch up)
akurender -> sim.render

The display will only update when glutPostRedisplay() is called.
Because _onPaint() and _onTimer can go out of they sync (paint is in the main glut loop and onTimer is set to a callback)
When the framerate drops (due to system variances) onTimer fluctuates slightly causing an extra render that the display doesn't know about and hence a judder.

The graphics update is in paint on a "fixed" timer where the sim.update -> sim.step can "sneak" in an extra frame because of the fluctuations that the display doesn't know about.

One fix is to ensure that _onTimer runs faster than SIM thereby ensuring the graphics card is always ahead of SIM even if this increase the number swaps.

In GL documentation each call to glutPostRedisplay (); will only cause the graphics to swapbuffers once - (ie you can call glutPostRedisplay() numerous times - its just a flag to swapbuffers

In onPaint() akurender could be called more than once whilst glutSwapBuffers does nothing because the glutPostRedisplay hasn't been called.

The update routines are finishing easily in time for GL. Increasing stuff in sim.step makes sense when the system is overloaded and is slowing down (ie too many physics, sprites, etc) but the bottleneck isn't likely to ever be the graphics display - You then take the hit and have a forced judder/glitch visually.

Another fix is to ensure each sim.step gets a render - (perhaps by calling sim.render from sim.step instead of GL Paint via akurender) - you'll get a chipmonk mode where visually everything speeds up to catch up rather than "cutting" with a perceived graphics jump (judder). Note: I haven't tried this theory yet..

Re: jerky / hitchy / jittery scrolling under Windows, iOS

by patrick » Fri Feb 24, 2012 8:18 pm

OK, so here's the latest. On Android, as Adam mentioned, there seems to be no clear way to get a constant framerate. We did a casual poll of existing games across the gamut and observed hitching in pretty much all of them. Unfortunately, we need to get back to our 1.0 feature track and are going to have to put the Android issue on hold for a bit. If in the mean time anyone in the community can figure out how to get a consistent framerate on Android please let us know. FWIW our tests on Android were *without* any Moai, just a basic surface rendering nothing at all. Still saw the framerate thrashing.

On Windows/GLUT... according to WGL, on our systems the GL interval rate defaults to 1 (60hz) . We did see much smoother performance by moving the call to AKUUpdate into _onPaint, calling glutPostRedisplay at the end of _onPaint and getting rid of the timer entirely. It seems GLUT's timer runs inconsistently, whereas simply relying on glutPostRedisplay from inside of the render callback produces a smooth 60hz. We tried adding a call to Sleep(1) in _onPaint to free up the CPU and immediately saw the hitching return. Interestingly, we saw tearing both with and without a fixed refresh interval. Clearing the interval allowed the tearing to 'skitter' over the window. Fixing the interval 'locked' the tearing into a fixed line on the screen.

All of the GLUT experiments were done using a fixed sim step of 1/60 with spin enabled. Even thought I was able to get smooth performance, I don't know if we're going to ship with a modified GLUT just yet. We still need to look at some other options, include Bach Le's Akuma host (which uses SDL). I should also mention that the reason we added the timer into the GLUT host was that the original host (back in the day) would hog the CPU.

So... wish I had better news. The silver lining is that, though annoying, the jittering/stuttering/hitching/juddering seems to exclusively be a host/timer/vsync problem. We're pretty confident at this time that there's nothing inside of Moai that causes these artifacts beyond the usual suspects (memory allocation in debug mode, garbage collection). We'll return to troubleshooting the hosts for this toward the end or just after the 1.0 launch. Right now the upshot seems to be:

1. Smooth performance with release builds on iOS devices.
2. Jittery performance on Android - gets better with variable timestep enabled. Similar performance to existing non-Moai games.
3. GLUT - better performance *without* using timer and updating sim in _onPaint (with fixed step on the sim). Trade off is host will hog the CPU.

  5. //////#include <GL.h> //not neede its in the glew
