Re: cycling between colors using MOAIProp2D:seekColor

by DanielWolf » Mon Oct 13, 2014 12:56 pm

Have a look at the RGB cube. (It only shows three sides, but you can imagine or google the rest )



You'll notice that all the saturated colors are one the surface. The more you move into the cube towards its core, the muddier the colors get.

I haven't tried it, but when using linear interpolation, Moai draws a straight 3-dimensional line from your start color to your end color. You want to prevent that line moving through the cube; you want it staying on its surface. So you'll have to introduce intermediate keyframes when moving from one side to another.

So, for instance, moving from red directly to cyan will move you right through the middle of the cube. Instead, add a magenta or yellow keyframe inbetween, and it should look just fine.

Let me know if it works for you!

Re: cycling between colors using MOAIProp2D:seekColor

by DanielWolf » Wed Oct 15, 2014 6:45 am

HSV and RGB work completely differently. In HSV, you can control the hue of a color (that is, its position on the rainbow) independently from its saturation and its relative brightness. So you can do a smooth rainbow cycle by just animating the H value.

In RGB mode, you're not so lucky. As you found out, using seekColor() to blend even between two fully saturated colors may result in ugly, grayish colors between. Nonetheless, you can achieve a beautiful, saturated color cycle using seekColor() alone. In fact, that's your only choice unless you want to do a lot of manual color conversion.

I guess my original post was a bit terse, so let me try again.

The color cube is a visual representation of the RGB color space. The three axes of the cube represent the R, G, and B values, respectively, each in the range 0..1. So every conceivable color has a corresponding point on or within that cube.
What you want is stay on the surface of the cube. That's where all the saturated colors are. The colors inside the cube are the muddy ones. In addition, you want to avoid the tree sides of the cube that aren't visible in my picture. These three sides contain the dark colors.

So let's say you want to blend smoothly from red through green to magenta. What seekColor() does is draw a straight 3-dimensional line from the start point to the end point. Imagine a straight line from red to green. That line will move along one of the far sides of the cubes; that's where the dark colors are. You don't want that. So you have to introduce an additional keyframe between the red one and the green one. Looking at the cube, yellow looks like a good fit: Moving from red to yellow will lead you along beautiful shades of orange, and moving from yellow to green will give you a lot of great lime colors.
Now let's move from green to magenta. Moving directly will take you right through the center of the cube -- that's where gray lives. So again, introduce a keyframe inbetween. One obvious choice is cyan. If you want a continuous cycle, you can now go back to red directly: There are nice shades of purple inbetween.

So your code might look like this:

Code: Select all
  1. MOAISim.openWindow ("test", 640, 480)

  2. local viewport = MOAIViewport.new ()

  3. viewport:setSize (640, 480)

  4.  

  5. local color = MOAIColor.new()

  6. color:setColor(1, 0, 0) -- red

  7. MOAIGfxDevice.setClearColor(color)

  8.  

  9. function threadFunc()

  10.     local action;

  11.     local alpha = 1;

  12.     local blendDuration = 5;

  13.    

  14.     while true do

  15.         action = color:seekColor(1, 1, 0, alpha, blendDuration) -- yellow

  16.         while action:isBusy() do coroutine:yield() end

  17.  

  18.         action = color:seekColor(0, 1, 0, alpha, blendDuration) -- green

  19.         while action:isBusy() do coroutine:yield() end

  20.  

  21.         action = color:seekColor(0, 1, 1, alpha, blendDuration) -- cyan

  22.         while action:isBusy() do coroutine:yield() end

  23.  

  24.         action = color:seekColor(1, 0, 1, alpha, blendDuration) -- magenta

  25.         while action:isBusy() do coroutine:yield() end

  26.  

  27.         action = color:seekColor(1, 0, 0, alpha, blendDuration) -- red

  28.         while action:isBusy() do coroutine:yield() end

  29.     end

  30. end

  31.  

  32. thread = MOAICoroutine.new()

  33. thread:run(threadFunc)

  34.  


Just as a note: For the sake of simplicity, I've used the colors at the corners of the cube only. But of course you're free to use any mixed colors, as long as you stay on the cube's surface.

Re: cycling between colors using MOAIProp2D:seekColor

by Vavius » Wed Oct 15, 2014 9:06 am

HSV is supported in C++ part of MOAIColor. I can expose it.

The only question is Lua API:
Code: Select all
  1. color.ATTR_H_COL

  2. color.ATTR_S_COL

  3. color.ATTR_V_COL

  4.  

  5. color:seekColorHSV(h, s, v, a, duration, easing)

  6. h, s, v, a = color:getColorHSV()

  7. color:setColorHSV(h, s, v, a)

  8.  

  9. h, s, v = MOAIColor.toHSV(r, g, b)

  10. r, g, b = MOAIColor.toRGB(h, s, v)



When you set RGB attr HSV values are updated automatically, and vice versa (just like ATTR_QUAT in MOAITransform).

Possible values:
H = [0, 360]; S = [0, 1]; V = [0, 1]
R = [0, 1]; G = [0, 1]; B = [0, 1]
