Features and Requirements for a MOAI 1.6 Stable Release

by ibisum » Fri Oct 02, 2015 9:49 am

I would like everyone to report on what they think is necessary for us to attain a stable release of MOAI 1.6.

As many of us know, 1.6 has been 'hanging fire' as a stable release for a long time now, and its time for us to nail down those features we absolutely must have in 1.6, as well as think about what we can put forward to the next version. The goal is to have a stable 1.6 that includes not just full working sources (out of the box - i.e. no hacking required to get builds for all platforms), but 'official' binary packages, as well as fully working samples and updated documentation.

From my perspective, I have only a few minor things to note - but will update this thread as more investigation occurs:

* MOAI 1.6 should be buildable, out of the box, for all of the supported platforms: Android, iOS, Windows, OSX, Linux. 1.6 should also be buildable and usable for raspberryPi.
* moaiutil should be stable and usable out of the box to create new MOAI host projects.
* 1.6 samples should all be updated to use the new 1.6 features - i.e. *zero* broken samples.
* MOAI 1.6 should be fully integrated with the CI build system, such that BINARY releases of the 1.6 SDK are available to those who want them, too.
* MOAI 1.6 documentation must be updated to reflect the reality of the 1.6 changes.
* Documentation/Tutorials on how to add a MOAI plugin to the system.
* Improved Joystick support.

?? Other features ??

This is something I need help with - identifying feature-branches that we might like to pull into 1.6. At the moment, my perception is (please correct me if I'm wrong) that there are a number of core developers who have 1.6 branches with new features in them - halfnelson, patrick, spacepluk etc. - and you guys should propose feature-branches/pull-requests that you think should be in the 1.6 release. We should nail these features down so that we can prepare the release in a timely fashion. I would think its feasible for us to have a 1.6 release within the next few weeks, but it depends entirely on what you, the community, think that 1.6 should include. For example, I have the opinion that we should *definitely* have SVG/vector support in 1.6 from patricks' branch - but this will take work to merge the changes, triage the bugs - if any - and prepare the feature for inclusion in a release. Are there other features like this that we should consider?

Please detail your thoughts on this now, so we can discuss it at the next MOAI Council (see here for details: feeling-abandoned-t3210/) meeting next week, and then coordinate our release schedules and come up with a plan for how we will get a stable, public, releasable 1.6 build done. ALSO: Please include a link to your repo/branch if you have done work on this already - its important for us in the MOAI Council to build a map of what work is done already, out there, and what we will need to do as well.

The branch that will be the target of our efforts to produce a releasable product: https://github.com/moai/moai-dev/tree/1.6-stable

IF there are other places that contain 1.6-qualified code, please list it - and understand that it is our intention to have these features available from the above repo (https://github.com/moai/moai-dev/tree/1.6-stable) ..

About binary releases - we'd actually like the community to hold off on releasing MOAI binaries for now - the idea is that we'll push to get the valuable changes into the main branch, and then will produce 'official' binary packages as part of the build server/CI setup that is almost working. Its definitely necessary for us to produce binaries - but we're trying to integrate things back into a more 'centralized' branch so that we don't lose the valuable development efforts - from this, we'll start producing binaries.
