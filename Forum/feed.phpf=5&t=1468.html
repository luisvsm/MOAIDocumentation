- A proper build tool and project templating tool. Copy and paste project template is not fun. We can take a look at Erlang's rebar. From creating project, downloading dependency to building and deployment, unit testing. Everything requires a simple command. Users can extend the tools and add more commands to it. Don't like command line? A GUI can be built on top of it.
- A library convention. Moai already has it. On the Android side, it's easy to include a library from another directory. I'm not sure about iOS. We just need a convention of how to organize (pure lua) addon library. So that a few lines in the project config file will add it to the project.
- A proper plugin interface. Hacking source to add third party plugins only work for you. We need to let other people replicate the process too. IMO, this can be done through templating the host's source code. Plugins will have to declare the codes they want to inject into the host so that a project can be built. With plugins, we can choose what we need and discard the rest.
- Native bridge. Both Objective C and Java can be called dynamically from Lua. Java has LuaJava, and there's ObjectiveC Lua bridge too. If it is bound to Moai, most of the problems of using platform specific libraries and codes will go away. The performance will not be as good as native binding but for most libraries, it will be fine.
- A good host template. The "official" one is an example (a good one even), it should be treated as such. People shouldn't deploy it and expect things to work.

Rather than building a community fork, I suggest that we build a community toolset (build tool, templates, plugin frameworks). They are totally separate from Moai. With a plugin interface, features can be added, removed at will without touching the source. For bug fixes, it is easy to make your own personal fork, create your own branch and merge the patches you want. Then you can point the above mentioned project tool to your own folder and build. That's the thing about git, everything is decentralized, you pick what you want from the community. With a common way to do things, if something works for you, it works for everyone. Bugfixes should be treated differently from feature enhancement.

I suggest this because I can tell from the forum that a lot of people have extremely inefficient workflow and bad project setup. Once again, the Moai SDK shows you examples, it's organized in a way that can show case multiple projects. Most people work on a single project at at time. We need to set a good example of what a typical game project looks like. Combine it with good tools to customize. I strongly suggest we take a look at rebar. It's one of the best project tool out there.
