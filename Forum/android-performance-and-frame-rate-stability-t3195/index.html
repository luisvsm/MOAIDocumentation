by JDance » Mon Aug 10, 2015 10:40 am

I should have written this a year ago but I'll try my best to remember.

Anyway my game was running silky smooth on iOS with 60 fps synced to the screen. On Android it just felt sloppy and uneven. First I tried to increase the performance by optimizing drawing and stuff like that, but I never got it to feel smooth. I suspected the game loop was causing some strange delays, and I researched all I could find on different ways of making an Android game loop that is smooth. Eventually I got it working great, and this is what I ended up with.

MOAISim: Remove all logic from here, just forward the step to lua, see https://github.com/JDaance/moai-dev/blo ... .cpp#L1047

LUA: Now - I found some good stackoverflow post about Android frame rates, and how the delta time reporting on Android is very spiky and way off. So this is key - in my lua code I take the average delta time from the last 5 frames.

Android will automatically cap the framerate to 60 fps, and the delta timings will be accurate because the spikes are averaged out. Its basically super simple. My game runs really smoothly on the devices I have tried. Especially compared to before, and also compared to other games.

Re: Android performance and frame rate stability

by Vavius » Mon Aug 10, 2015 3:37 pm

I want to make sure that the latest MOAI version have those changes for smooth framerates.

Android host:
Putting Moai.update() near Moai.render() is an old trick (IIRC credits to Makoto).
Maybe a more correct solution is to use Choreographer.FrameCallback (API level 16 - Android 4.1 the first jelly bean).
Actually, both ways is a vsync. The same as displayLink on iOS.

Current 1.6 and develop android hosts in the main repo uses an old-fashioned Runnable. Incredibly inaccurate. This is the main source of spikes.

MOAISim:
The exact same behavior can be achieved with loop flags: SIM_LOOP_ALLOW_BOOST and mBoostThreshold = 0. It will step once each frame using real elapsed time as delta.

Lua:
This is the interesting part. Can you explain a bit more where and how do you use this averaged delta time?
Seems like MOAI uses clock_gettime( CLOCK_MONOTONIC ) on android. This is a low-level timer with ns resolution frequently used for profiling and stuff. I doubt it can be the source for visible glitches.

I thought that using exact delta time should give better visual results. The idea is to scale all animations with the actual elapsed time. The proper way to get dt in MOAI is to use return result from coroutine.yield(), but not values from MOAISim. All other actions like seekLoc are scaled internally by engine.
Of course, above is true only when the timer is accurate.

This is the stack overflow post that inspired me to use a mean: http://stackoverflow.com/a/10730291/1613430
Check out the graphs there. My take from reading here and there was that no matter what resolution (ns or ms) the reported time was just bad.

I do use coroutine.yield() for the delta time. I never plotted the reported delta times myself so its possible that they are not as bad as in that graph.. but my game sure is smooth using mean value. I just average between the last 5 reported delta times.

Lua mean class:
Code: Select all
  1.  

  2. ---------------------------------------------------------------------------

  3. -- MEAN

  4. ---------------------------------------------------------------------------

  5.  

  6. local C = {}

  7. C.__index = C

  8. Mean = C

  9.  

  10. setmetatable(C, {

  11.         __call = function(class_tbl, size)

  12.                 local obj = {

  13.                         size = size,

  14.                         t = {}

  15.                 }

  16.                 setmetatable(obj, C)

  17.                 return obj

  18.         end

  19. })

  20.  

  21. function C:is_a(class)

  22.         return getmetatable(self) == class

  23. end

  24.  

  25. ---------------------------------------------------------------------------

  26. -- PUBLIC

  27. ---------------------------------------------------------------------------

  28.  

  29. function C:add(val)

  30.         local t = self.t

  31.         table.insert(t, val)

  32.         while #t > self.size do

  33.                 table.remove(t, 1)

  34.         end

  35.         self.cache = nil

  36. end

  37.  

  38. function C:getMean()

  39.         local t = self.t

  40.         if not self.cache then

  41.                 local mean = nil

  42.                 for i = 1, #t do

  43.                         mean = (mean or 0) + t[i]/#t

  44.                 end

  45.                 self.cache = mean

  46.         end

  47.         return self.cache

  48. end

  49.  

  50. function C:__tostring()

  51.         local mean = self:getMean() or "nil"

  52.         return "MEAN([" .. table.join(self.t, " ") .. "] = " .. mean .. ")"

  53. end

  54.  

  55. ---------------------------------------------------------------------------

  56.  



Usage is just something like
Code: Select all
  1. local mean = Mean(5)

  2. local rawDelta = 0

  3. while gameLoop

  4.  mean:add(rawDelta)

  5.  local delta = mean:getMean()

  6.  

  7.  stepGame(delta)

  8.  

  9.  rawDelta = coroutine.yield()

  10. end

  11.  


It will discard older values when there are 5 or more

I realise now that my game is a bit of a special case since I use no actions at all. I have physics code that runs on a fixed time step, but I interpolate between physics frames depending on time step. All animations and stuff are run from lua with a scene graph node:update(dt) kind of system.

If you use actions I guess the delta time smoothing (mean value) would have to be calculated in MOAISim or in the Android host.

iOS
