RE: Rocket Lobster Tutorial - A thread for every explosion?

by mervills.email » Mon Dec 19, 2011 2:46 pm

Hello eveyone.

I had a question regarding the Rocket Lobster Tutorial, specifically about this block of code: 

function makeExplosion ( x, y, size )



	local explosion = MOAIProp2D.new ()

	explosion:setDeck ( explosionGfx )

	explosion:setLoc ( x, y )

	explosion.size = size

	layer:insertProp ( explosion )



----------------



	function explosion:checkHit ( prop )



		local x1, y1 = self:getLoc ()

		local x2, y2 = prop:getLoc ()

		return distance ( x1, y1, x2, y2 ) <= self.size

	end



----------------



	function explosion:main ()



		for i = 1, self.size do

			self:setFrame ( -i, -i, i, i )

			for rocket in pairs ( enemyRockets ) do

				if self:checkHit ( rocket ) then

					rocket:explode ( 64 )

				end

			end

			coroutine.yield ()

		end

		layer:removeProp ( self )

	end



	explosion.thread = MOAIThread.new ()

	explosion.thread:run ( explosion.main, explosion )

end


Coming from a background of XNA, having each explosion run in a dedicated thread seems like a bad idea. Especially in a larger project where lots of objects would be running in this fashion. My first thought would be to have a single thread that triggers update() functions on all the objects registered to it.

Basically my question boils down too: Is it considered good practice to create a new thread for every new object as is done in the Rocket Lobster Tutorial?

I'm just learning the concepts of multi-threading in general, so please excuse me if I've overlooked something obvious. 

Thanks!

- Mervill

RE: Rocket Lobster Tutorial - A thread for every explosion?

by bonozogames » Mon Dec 19, 2011 3:07 pm

I believe these "threads" are actually Lua "coroutines", and not "pure" multi-threads.  As in, the over head is primarily the stack pushing/popping and perhaps the creating/removing, but not necessarily the amount of threads the major serverity.

For my current apps, I myself, am not using a MOAIThread per object, but I have done so, in Lua, in the past, for another game. One of the supposed "benefits" is to help control how often objects are ticked;  not all need to be every game loop, or even every other game loop, some could more infrequent (ex. "sleep(1000)" instead of "coroutine.yield()").  But alas, an update service can also manage this well, just respecting the time, and, like you pointed out, still most likely more efficient CPU-cycle wise.

In theory, I believe a thread per object is a better architecture, but as a practical matter, I believe update services have less overhead.

There's most likely a lot more to this subject that is just not coming to mind at the moment.

RE: Rocket Lobster Tutorial - A thread for every explosion?

by bonozogames » Tue Dec 20, 2011 3:17 pm

In thinking about this more, the thread (i.e. coroutine) per explosion is a good thing.  Remember, they are not threads, they are scheduled tasks.  They are basically already added to an "update loop", and thus you don't have to write one.  I came to this conclusion because recommending an update service was more for a C/C++ implementation;  I don't see the big win in writing an update service in Lua, over Lua coroutines.  I just remember thinking in a previous game I did not want 100s of coroutines, but I cannot think of the reason why, currently.

RE: Rocket Lobster Tutorial - A thread for every explosion?

by patrick » Thu Dec 22, 2011 9:54 pm

That's exactly right. The per-coroutine overhead is very low. Having a coroutine per object is a great way to manage the object's update loop and state machine. MOAIThread just integrates Lua's coroutines with the Moai action tree.

(Would people be happier is the name was MOAICoroutine instead of MOAIThread?)
