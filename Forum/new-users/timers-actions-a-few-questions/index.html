Yes, no and good idea.

The start/stop is just an alias for adding and removing the action as a child. Have thought about doing away with them to force explicit add/remove semantics in the action tree, but will probably keep the status quo for now. Returning the action from all the operations to allow method chaining is a great idea. I added it to our backlog, but we may not get to it for a bit. If you beat us to the punch, please send a pull request.

Regarding MOAIInstanceEventSource, there are two versions of the function, but only the first is implemented (and I don't get the point of the second one). 


//================================================================//
// MOAIInstanceEventSource lua
//================================================================//

//----------------------------------------------------------------//
/**	@name	setListener
	@text	Sets a listener callback for a given event ID. It is up
			to individual classes to declare their event IDs. This function
			has two forms: one for instances and one for singletons. When
			using setListener with a singleton (such as MOAISim), simply
			omit the first parameter (self).

	@overload

		@in		MOAIInstanceEventSource self
		@in		number eventID			The ID of the event.
		@opt	function callback		The callback to be called when the object emits the event.
							Default value is nil.
		@out	MOAIInstanceEventSource		self

	@overload

		@in		number eventID			The ID of the event.
		@opt	function callback		The callback to be called when the object emits the event.
 							Default value is nil.
		@out	nil
*/
int MOAIInstanceEventSource::_setListener ( lua_State* L ) {
	MOAI_LUA_SETUP ( MOAIInstanceEventSource, "UN" );

	self->SetListener ( state, 2 );
	state.CopyToTop( 1 );

	return 1;
}
