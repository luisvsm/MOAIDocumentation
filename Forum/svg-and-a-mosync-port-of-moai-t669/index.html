Re: SVG and a Mosync port of Moai

by ankharlyn » Fri Apr 26, 2013 5:30 pm

I would suggest avoiding Cairo. It's decent if *all* you want to do is render some SVGs to raster of a certain resolution, but it's not even close to fast enough to re-raster multiple SVGs per frame.

I'd suggest looking at libraries that sit on top of OpenVG, like ShivaVG. I've done some tests and, unlike Cairo (and many other SVG renderers) ShivaVG/OpenVG is fast enough to do "dynamic" SVG rendering (i.e., where you might want to change the path data dynamically during simulation).

Re: SVG and a Mosync port of Moai

by ankharlyn » Sat Apr 27, 2013 9:22 pm

ibisum wrote:Very interesting .. I'm wondering if ShivaVG can be shimmed into the MOAI client in some sensible way ..


Should be fairly straight forward, as after ShivaVG processes your VG commands (parsed in from SVG or given manually) it's just giving you a bitmap texture, essentially. At least, from what I can recall. I may be misremembering. If that is in fact the case, it shouldn't be that hard to set up the VG-"textures" in such a way that you can control how often they redraw etc and then otherwise handle them like normal textures.

I'll have to see if I can dredge up my test bed for it (written on top of SFML) and see how it worked again. In the testbed, this effect was somewhat "hidden" as I had it set up to re-rasterize the vector data every single frame, so zooming in and such was seamless as in programs like Illustrator/Inkscape.

Also I guess it doesn't need to be ShivaVG though from what I recall it has a favorable license. Anything that works on top of OpenVG will be fast enough.

The reason I had this requirement for such speed (took me years to find something to do it properly) is because I was planning to use physics feedback on path control points to make them jiggle/wobble and such, as well as using the control points as "vertices" for the purpose of skinning. Or even just set up animation for it, which of course requires re-rasterizing a lot of paths quite quickly.

OpenVG based libraries are the only ones I found that are able to handle it, aside from QT's SVG rendering, but I discarded that because it was so abstracted away you couldn't really alter path data at run time.
