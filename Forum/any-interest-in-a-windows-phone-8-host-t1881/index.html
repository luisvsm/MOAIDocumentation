[CONTRIB] Any interest in a windows phone 8 host?

by halfnelson » Fri Jun 07, 2013 12:55 pm

Seems windows phone 8 is growing in market share http://www.kantar.com/media/mobile/030613-smartphone-sales-data-us/ and is outselling blackberry, is there any interest in getting moai to run on windows phone 8?

I have come across a great little library called ANGLE which is what chrome and firefox use for the webgl implementations on windows. It translates opengles 2.0 calls to directx 11.1 (no opengl on win phone). The QT project guys in the last couple of days got this to build for winrt which should help with the port.

I am having a little spike at the moment, and although it looks harder than getting the html5/js host running (too much c++ not enough javascript ), it certainly looks possible. I would bank that the guys at marmalade use the same lib for their new windows phone 8 support [just looking at their known issues, i'd say that is true] so it is definitely doable. Our job is made easier by the fact that zipline abstract away the opengl implementation and only use a subset.

scottwb wrote:I would be interested

In terms of priority, my list would be:

- web/HTML5/Facebook
- Mac (app store)
- Blackberry
- Windows 8

- Web/Html5 : Done, see pull request
- Mac (app store): what are the limitations of our current mac host? what needs fixing?
- Blackberry : The zipline guys are already working on this (BB10)
- Windows 8: is this "windows 8 store" or "windows phone 8"? They are slightly different in annoying ways. I was thinking since this is mainly a mobile framework, we should focus on the mobile space first. Windows 8 Store support might be easier and a stepping stone though.

ezraanderson wrote: but why not just a directx host?

Well it would be a directx host since we don't really have a choice. The AngleProject opengl es emulation layer would be a good fit since ziplines gfx abstraction is based on opengl anyway: zldrawarray->gldrawarray. Using angle should give us a simple
Code: Select all
  1.  

  2. #ifdef WINDOWS_PHONE

  3.    #include "angle/gles2.h"

  4. #endif

  5.  


allowing the least amount of changes to libmoai meaning more likely to stay up to date.
