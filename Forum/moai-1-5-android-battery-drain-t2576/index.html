Re: MOAI 1.5 Android Battery Drain [cause found]

by Motig » Wed Mar 19, 2014 10:30 am

Hey everyone, I found the cause of the battery drain.

Code: Select all
  1. MOAIUntzSystem.initialize(44100, 1024)



I've done multiple tests now and the moment I put this back in the drain started happening again. This is without actually doing anything else untz related, just initializing it. Not sure where to go from here now though. Also it is possible this is also happening on older version of MOAI, I've never had untz in my older builds on previous versions, and as it doesn't show up in the battery usage stats (in older android versions) it's possible people have not been noticing it.

*Edit: Could it be that untz is never being suspended/called?

Re: MOAI 1.5 Android Battery Drain [cause found]

by halfnelson » Wed Mar 19, 2014 2:36 pm

Ahh, this makes sense, the module system was refactored in patricks branch, I don't remember seeing any special calls to modules when AKUPause is called where in the old system they would pause untz.

EDIT: Yep, calls to MOAIUntzSystem::Resume () and MOAIUntzSystem::Suspend () are gone. They used to happen on pause. Should be easy enough to fix. I am guessing that the module system needs a pause and resume callback incase future modules need to stop what they are doing when the app goes into the background.

Here is where the call used to be:
Code: Select all
  1.  

  2. extern "C" void Java_com_ziplinegames_moai_Moai_AKUPause ( JNIEnv* env, jclass obj, jboolean paused ) {

  3.                 AKUPause ( paused );

  4.                 if ( paused ) {

  5. #if MOAI_WITH_UNTZ

  6.                         AKUUntzSuspend ();

  7. #endif

  8.                 } else {

  9. #if MOAI_WITH_UNTZ

  10.                         AKUUntzResume ();

  11. #endif

  12.                 }

  13.         }

  14.  

  15.
