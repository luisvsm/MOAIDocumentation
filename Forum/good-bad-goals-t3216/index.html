~
Could everyone post a quick list, keep it brief and to the point, we are trying to figure some stuff out based on peoples opinions of moai.

  • What is good about MOAI
  • Whats is bad about MOAI
  • What Goals should we focus on for improving moai


Goals:
  • 1. Better tutorials
  • 2. Better landing page for Moai
  • 3. Clear current features
  • 4. Clear upcoming feature
  • 5. Small but full game samples
  • 6. A clear defined road map
  • 7. Communicate bugs on a branch
  • 8. Show case of games and links to these games
  • 9. A clear IDE for navigating the samples, maybe zerobrane pre-packages with a moai-bin, relative paths, and a way to step through the samples.
  • 10. A clear guide line to market/promote Moai
  • 11. Easy build system to make android & iphone hosts
  • 12. monetization plan for patrick to start making income from moai
  • 13. Chose vs2012 for windows, and deprecate older versions
  • 14. needs way more shader examples.

Bad:
  • 1. People think Moai is too hard
  • 2. People think Moai is only for professionals
  • 3. Getting MOAI out of the box and working correctly on all platforms is difficult
  • 4. not enough developers use MOAI
  • 5. The successful developers are unknown, and don't interacted with the community
  • 6. fractured community, everyone focusing on their own branch/project
  • 7. unclear future/road-map/goals
  • 8. confusion website.
  • 9. the default android host, seems crapy

Good:
  • 1. Good
  • 2. amazing & powerful
  • 3. open source
  • 4. supports all platforms
  • 5. lots of samples
  • 6. good api documentation
  • 7. fast and efficient for prototyping new ideas
  • 8. 2D & 3D
  • 9. lua based api

Re: Good & Bad & Goals

by halfnelson » Sat Oct 03, 2015 2:08 am

Good:
* Free
* Open Source
* Clean well structured code
* Fast
* Standout features are:
* Easy to extend in c++ land
* Powerful Attribute binding/dependency system
* Desktop hosts are first class citizens

Bad:
* Docs out of date
* No stable releases with clear documentation on changes
* Plugin infrastructure not quite there yet
* Bugs floating around due to lack of users probably
* Default hosts can be improved (I suggest grabbing stirfire host for desktops and updating android host with ezra's/vavius's fixes)

Goals:
* Get new users
* Make it clear why you would use MOAI over Corona, Marmalade, Gideros, Libgdx, or Godot
* Promote the node/dependency system as an advantage
* Bundle a get started sdk release that includes everything to get going (ZBS, android stuff, texturepacker, flower lib, example project setup ready to go)
* Integrate something like vavius' qt host that makes it easier to debug/prototype mobile apps using desktop host.
* Integrate stirfire host as production ready cross platform sdl host
* Updated docs where users can contrib (like http://danielswolf.github.io/moai-dev/d ... cmake.html)
* not vs2012 but vs2013 should be default (it is available for free as community edition)
* Patrick $$$ = I propose that the ad network plugins get pulled out of main repo (except maybe one) and can be purchased for a modest fee (that includes upgrades for x months) from some kind of plugin store (see corona) alternatively you could give the plugins away for free but bump it up if they make more than $x per year. Other option is paid support (you pay and get priority on feature implementation request/bug fixes) like other open source projects.

Re: Good & Bad & Goals

by ibisum » Wed Oct 07, 2015 12:27 pm

MOAI-Good: Fast, pleasing environment in which to build an application. Excellent license, if you grok it. Comfortable API, SDK for describing most any form of user interface. Artist-/Platform- friendly assumptions about the project goals. A form of Cross-Platform development that promotes competency on all platforms. Easy to set up a comfortable environment for coding, because: "$ moai ./main.lua" .. which will run pretty well the same on all supported platforms. (iOS, Android, Linux, Windows, OSX, &etc.) Once you see your app running well on everything, the platform wars no longer appeal, and that is good for sanity. Codebase is easy to understand once you've generated the docs. Samples are easy to step through and get the basic concepts. Lua. C,Lua,CMake,Xcode,GLES,VistualStudio,Linux,Android,iOS,OSX,&etc. experience is useful, but you can just do everything in Lua if you want, as well. Nobody else uses it, so the power is all yours.

MOAI-Bad: "Business guys" can't grok it. A form of Cross-Platform development that is difficult if you are unwilling to assume responsibility on all platforms. Requires a 'long runway' before you get things running in equilibrium on all platforms. Requires platform-specific competence. C,C++,Java,Objective-C,Lua,CMake,Xcode,VistualStudio,Linux,Android,iOS,OSX,&etc. experience will be required (or at least recommended) before you can do anything in a pure Lua environment. Nobody else uses it, so the problems are all yours.

MOAI-Improvement-Goals: Immediate stable forward progress through two or three public releases of the SDK in order to address: host-creation, documentation, and testing of the current codebase, synchronicity of the MOAI API across all platforms. Short-term Goal: MOAI IDE, GUI Framework advances, Host-OS features. Long-term Goal: establishment of MOAI as a viable alternative to pure-native development for the sustained creation of amazing user interfaces, games, and other forms of application.
