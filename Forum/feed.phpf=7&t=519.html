]]> 2012-02-24T20:18:35+00:00 2012-02-24T20:18:35+00:00 /forums/viewtopic.php?t=519&p=2525#p2525 <![CDATA[Re: jerky / hitchy / jittery scrolling under Windows, iOS]]>
On Windows/GLUT... according to WGL, on our systems the GL interval rate defaults to 1 (60hz) . We did see much smoother performance by moving the call to AKUUpdate into _onPaint, calling glutPostRedisplay at the end of _onPaint and getting rid of the timer entirely. It seems GLUT's timer runs inconsistently, whereas simply relying on glutPostRedisplay from inside of the render callback produces a smooth 60hz. We tried adding a call to Sleep(1) in _onPaint to free up the CPU and immediately saw the hitching return. Interestingly, we saw tearing both with and without a fixed refresh interval. Clearing the interval allowed the tearing to 'skitter' over the window. Fixing the interval 'locked' the tearing into a fixed line on the screen.

All of the GLUT experiments were done using a fixed sim step of 1/60 with spin enabled. Even thought I was able to get smooth performance, I don't know if we're going to ship with a modified GLUT just yet. We still need to look at some other options, include Bach Le's Akuma host (which uses SDL). I should also mention that the reason we added the timer into the GLUT host was that the original host (back in the day) would hog the CPU.

So... wish I had better news. The silver lining is that, though annoying, the jittering/stuttering/hitching/juddering seems to exclusively be a host/timer/vsync problem. We're pretty confident at this time that there's nothing inside of Moai that causes these artifacts beyond the usual suspects (memory allocation in debug mode, garbage collection). We'll return to troubleshooting the hosts for this toward the end or just after the 1.0 launch. Right now the upshot seems to be:

1. Smooth performance with release builds on iOS devices.
2. Jittery performance on Android - gets better with variable timestep enabled. Similar performance to existing non-Moai games.
3. GLUT - better performance *without* using timer and updating sim in _onPaint (with fixed step on the sim). Trade off is host will hog the CPU.

The flows are:
akuupdate -> sim.update -> sim.step (which is where all the sim flags are for variable frame rate - these work by looping back into sim.step to try and catch up)
akurender -> sim.render

The display will only update when glutPostRedisplay() is called.
Because _onPaint() and _onTimer can go out of they sync (paint is in the main glut loop and onTimer is set to a callback)
When the framerate drops (due to system variances) onTimer fluctuates slightly causing an extra render that the display doesn't know about and hence a judder.

The graphics update is in paint on a "fixed" timer where the sim.update -> sim.step can "sneak" in an extra frame because of the fluctuations that the display doesn't know about.

One fix is to ensure that _onTimer runs faster than SIM thereby ensuring the graphics card is always ahead of SIM even if this increase the number swaps.

In GL documentation each call to glutPostRedisplay (); will only cause the graphics to swapbuffers once - (ie you can call glutPostRedisplay() numerous times - its just a flag to swapbuffers
