RurouniGPL wrote:
Does the Spine library work with all the features of the current version of the software?
I use it in production. It can be updated easily to reflect current feature set of official C-runtime. Here is a glimpse on what's supported and what's not:

Supported:
  • Lua bindings. Loading of json skeleton format and spine texture atlas
  • Skins, animation tracks, setters for attachments per slot, animation events - start, stop, custom events from editor with parameters
  • Overriding animations with MOAI classes - each bone can be represented as MOAITransform (with hierarchy), so you can bind any MOAIProp to spine bones and vice versa (driving bones with MOAIAnim). Slots can be represented as MOAIColor (for color animations)
  • Mesh and skinned mesh attachments

Not supported:
  • Inverse-kinematics. Don't need this right now, but I'll add this if you want
  • Low-level bindings. Some features may be missing from lua-bindings. Like updating skeleton manually (I've solved it by adding step() function to MOAIAction, since it's very convenient for other actions too). If you find something missing - please drop me a mail or create an issue on github.
  • Missing API for manual creation of spine objects: skeletons, bones, slots, attachments
