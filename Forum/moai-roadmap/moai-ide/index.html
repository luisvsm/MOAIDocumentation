How important is this to people? I think a Lua debugger is undeniably in our future. There's a case to be made for a cross platform IDE that has a built-in simulator and some kind of app bundling utility.

Lua debugging would be a clear win, and a simulator would be useful for anyone developing for hardware specific inputs sensors. I'm less sure about app bundling tools. I don't know if trying to take Xcode, Eclipse and Visual Studio out of the picture is a great idea. It's certainly possible, but would we really be helping in the long run, or just obfuscating something that should be understood?

That raises another question: one way we could go with a debugger/simulator combo would be through an Eclipse plug-in. I've seen some very nice Eclipse plug-ins and some very… not nice Eclipse plug-ins. As an IDE development framework Eclipse is pretty slick, and it's also cross platform, which I like.

Beta users, early adopters, what say you?

Post edited 11:16 pm – July 13, 2011 by James Rhodes

Hi everyone,

I'm currently working on an IDE for MOAI so I thought I'd outline the current debugging structure that I'm working on.  Feedback on this is greatly appreciated.

Essentially when the MOAI engine starts, it will load an extension known as moai-harness, which provides a socket-based debugging interface to which JSON commands can be sent and retrieved.  The IDE takes advantage of this and connects to the MOAI engine right after starting it (if the engine loads harness, it will wait until the IDE connects before continuing execution of the game).

Once the IDE is connected, it'll send and receive JSON messages over this interface, allowing the IDE to control the engine.  This design has a few distinct advantages as far as I can see:

  • It allows for remote debugging of MOAI whether the engine be running on PC, Mac or mobile devices.
  • Even when debugging on a mobile device, the full set of debugging functionality will still be available since it is all sent over standard network sockets.

I think the parity of being able to do full debugging while running on a target device is something that's not necessarily available elsewhere.

Suggestions to this system are of course welcome.

James Rhodes sent me an email earlier today asking me to open up the moai-ide repository. James has been working on this on and off for a while now. The engineering decisions, choice of technology and framework (mono) are all his – and I think they are quite good.

I do want to stress that this isn't intended to be the "official" Moai IDE. As much as we like what James has done we simply don't have the bandwidth to maintain the IDE project internally. I also appreciate that there may be other technical approaches that could work just as well.

At this time I'm hoping a community will grow around James' project to help him maintain and extend it. If you're interested in the IDE and don't mind digging in a bit please check it out at:

git@github.com:moai/moai-ide.git

I'll let James take it from here.

Hi everyone,

I just finished getting a build server set up for the Moai IDE, you can access the 32-bit and 64-bit builds from http://build.redpointsoftware……oai%20IDE/.  Due to the way the code editing control works, if you're running 64-bit Windows you need to use the 64-bit version of the IDE (a pain I know, but hopefully the code editing control will be replaced soon).

If you'd like to help out with the IDE development, let me know; the current list of issues / features that need to be addressed are on the GitHub page (https://github.com/moai/moai-ide/issues).

Right now the top priority is finding an alternative code editing control which will work on Mac and Linux.  This is the only aspect on the IDE that's really holding back it's deployment to those platforms, so if anyone has any suggestions let me know or even better, fork and replace the code editing control yourself if you have the time.

Regards, James.
