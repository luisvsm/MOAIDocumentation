Pixel Animation Morphing

by naturally » Sat Dec 07, 2013 3:37 am

Been fascinated by this tech called "DMorph":

DMorph: http://www.youtube.com/watch?v=x15Ir-rplT8
The Executive: http://www.youtube.com/watch?v=H2wZsgePIz0

Something like this would drastically cut down on the amount of sprites I have to draw by hand, or just give more frames to work with for better animations. After a brief exchange with RiverMan, got this juicy tidbit:

The algorithm is similar to morphing software used for movies like Terminator 2, but with added flexibility and optimization. You can read up on the general approach here: http://en.wikipedia.org/wiki/Morphing

Our implementation runs in real-time in-game. It doesn't generate a spritesheet, but rather takes an existing spritesheet and makes it animate smoothly.

Basically the idea is to set up a polygon mesh on the first frame, and then copy that mesh to the second frame, but with the vertices moved around to reflect the change in shape and position of different elements of the sprite. Your frame number then becomes a floating point value that is used to interpolate between these two meshes. To render it, you multitexture both the first and second frame onto the interpolated mesh, using a mix() shader to adjust how much each texture is showing through based on the frame number.


Something like this would be a fantastic addition to MOAI; something that, perhaps, creates a TileDeck2D out of two or more Textures, matching user meshes (?) and a desired frame count.

Can anyone offer a more details and thoughts on how this might be achieved?

This could very well be something worthy of a bounty...

  6.   - Without shape hints

  7.       Probably all work should be done in fragment shader: for each pixel march thru both textures to find out edges of the image and interpolate depending on the distance to the edges from the current pixel.

Re: Pixel Animation Morphing

by nosheet » Mon Dec 09, 2013 1:29 pm

I am not sure about pixel morphing, this looks to me much easier to do using animated mesh.

As @JDance said you should create first a custom mesh, ideally in some 3D program. Lets call it mesh1 - this will represent your character on animation frame 1 (start position)
Then for the end position (ie. Frame 2) you should create another version of the same mesh, moving vertices into new positions. Ideally you don't want to add or remove vertices, just move them around.

The trick is to store both positions for each vertex in MOAIMesh, and then create vertex shader that will read both, and then interpolate between them, using the parameter between 0 and 1 that you forward as a float uniform to the shader ( 0 = frame1, 1 = frame2, and everything in between will tween vertices).


That would mean that you should create MOAIMesh in such a way that the MOAIVertexFormat has two sets of vertex coordinates.
That means that you should declare index twice:
Code: Select all
  1.  

  2. vertex_format:declareCoord ( 1, MOAIVertexFormat.GL_FLOAT, 3 )

  3. vertex_format:declareCoord ( 2, MOAIVertexFormat.GL_FLOAT, 3 ).

  4.  


So then later you should try to write your vertex format in following fashion:
vertex1_frame_1_xyz
vertex1_frame_2_xyz
vertex1_uv
vertex1_normal_xyz (if you need to peform normal-dependent stuff such as lighting)
vertex1_color (if you need to additionally tint themes)

You should also set the vertex attribute two times for the shader, ie:
Code: Select all
  1.  

  2.  shader:setVertexAttribute( 1, "vertex_pos_0" )

  3.  shader:setVertexAttribute( 2, "vertex_pos_1" )

  4.  shader:setVertexAttribute( 3, "vertex_uv" )

  5. etc...

  6.  



and then don't forget to add a variable that you can change from Lua code, that will tell shader
how much to interpolate (ie. 0-1):
Code: Select all
  1.  

  2.  shader:declareUniformFloat( NNN, "light_range", MOAIShader.UNIFORM_FLOAT )

  3.  


where NNN depends on how many unfiroms you declare before this line...
Hope this gives you the right direction

Re: Pixel Animation Morphing

by Vavius » Mon Nov 24, 2014 2:32 pm

RurouniGPL wrote:Does the Spine library work with all the features of the current version of the software?
I use it in production. It can be updated easily to reflect current feature set of official C-runtime. Here is a glimpse on what's supported and what's not:

Supported:
  • Lua bindings. Loading of json skeleton format and spine texture atlas
  • Skins, animation tracks, setters for attachments per slot, animation events - start, stop, custom events from editor with parameters
  • Overriding animations with MOAI classes - each bone can be represented as MOAITransform (with hierarchy), so you can bind any MOAIProp to spine bones and vice versa (driving bones with MOAIAnim). Slots can be represented as MOAIColor (for color animations)
  • Mesh and skinned mesh attachments

Not supported:
  • Inverse-kinematics. Don't need this right now, but I'll add this if you want
  • Low-level bindings. Some features may be missing from lua-bindings. Like updating skeleton manually (I've solved it by adding step() function to MOAIAction, since it's very convenient for other actions too). If you find something missing - please drop me a mail or create an issue on github.
  • Missing API for manual creation of spine objects: skeletons, bones, slots, attachments
