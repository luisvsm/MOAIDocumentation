Re: In Moai "1.5", Lua errors no longer display the Lua stac

by Vavius » Wed Feb 05, 2014 7:54 am

Now there is no default error handler.
You can set one from host via C callback function (with AKUSetFunc_ErrorTraceback) or call MOAISim.setTraceback() from lua.
Lua traceback function can look like this:
Code: Select all
  1. function(err) print(debug.traceback(err, 1)) end



However, I've had problems using lua side traceback with included host (sdl or glut, don't remember) - it wasn't printing anything, but it works fine from my Qt host. So it can be a problem with included host.

by thebullno1 » Sun Apr 06, 2014 12:47 am

I just read the code AKUSetFunc_ErrorTraceback doesn't do anything.
I think, as an optimization, when no stacktrace function is registered, Moai will use an unprotected call: https://github.com/moai/moai-dev/blob/master/src/moai-core/MOAILuaState.cpp#L171
To get a stack trace, you need to set the trace function before you crash. Which means if you start by running one script and that script crashes, you won't get any trace.

Try adding this before you do AKURunScript in your host:

Code: Select all
  1. AKURunString("MOAISim.setTraceback(debug.traceback)")

by Vavius » Sun Apr 06, 2014 2:28 pm

thebullno1 wrote:debug.traceback is sufficient as a traceback function. It is supposed to return a string, not printing things out.
In this particular case anything is sufficient as a traceback function, since MOAI checks if traceback is set and only then performs pcall.

MOAI doesn't handle any output from traceback function, but prints it's own stack trace from C++: state.PrintStackTrace ( ZLLog::CONSOLE, 0 );

This is why there is multiple traceback messages. If we want custom message, then we should print it...

Re: In Moai "1.5", Lua errors no longer display the Lua stac

by thebullno1 » Thu Apr 17, 2014 10:42 am

You get a panic because the code that sets the callback crashes. Before you set a callback, there's no callback so the code that sets the callback is called in an unsafe manner which panics and die.

Here's what I do in my host:

AKURunString("MOAISim.setTraceback(function() end)");
AKURunScript("main.lua");

It makes sure that main.lua is called safely.
The code in AKURunString should be minimal and absolutely cannot crash.

by JDance » Fri Apr 18, 2014 12:46 pm

I think the correct behaviour would be to have a default stack trace function - why would you not want to have one? BUT you can override it with your own, and then you get to decide if you want to print something or not - maybe you just want to upload the trace.

So
1. Default stack trace with lua_pcall is printed if nothing is done - all run from MOAILuaRuntime, no AKURunString
2. MOAISim.setStackTrace overrides the default stack trace function with your own function - at this point the engine does not print any stack trace or anything and its all up to you

I think that is a very flexible solution. Maybe someone insists on being able to skip pcall? Maybe there should be an AKU function that the host can call to disable pcall then. Pcall with stack trace is a very sensible default IMHO.
