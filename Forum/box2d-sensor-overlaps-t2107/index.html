I'm writing a plaformer with a hybrid collision detection system. For the main dude, I'm using sensor fixtures and manually setting the body position with setTransform, so that I can handle all the movement and just use the collision detection aspect of Box2D without it resolving all the physics for me. Other objects I'll probably make standard objects (non-sensor) and Box2D can do what it likes!

Bit stumped though, on how to get the overlap deltas on a collision (for example, the dude landing on a platform). Re: Box2D, sensor overlaps

by Vavius » Wed Oct 16, 2013 9:06 am

Sensors do not generate PreSolve and PostSolve callbacks. Impulse is only known in postSolve.
But for your purpose of handling the Main Dude collisions manually just the impulse in not enough. You also need the "separation" value - it is the overlap amount. Box2d use this values internally.

Definitely, it is not possible in Lua side. And even in C++ side it is not trivial - box2d's default contact listener does not expose separation values.

When bodies overlap they first get pushed away by some percent each step (without applying velocity) and final collision impulse is accumulated (this is called TOI solver - time of impact) and affect the velocity. You can make two rigid bodies and nullify their velocity in each postSolve step - they will be pushed away just fine, I believe results will be just like setting restitution to 0.

Having main character physics out of all other physics system is good approach and it is used in every AAA-title (Erin Catto have recommended it on box2d forums). But, to make such a system you should learn some box2d internals and implement custom collision solver aside. It is hard and time-consuming. It's not worth for simple 2d platformer game without complex skeletal animated characters with some Inverse kinematics solvers and such...

So, I would recommend this setup for the main character:
1) Rectangular rigid body with circle at the bottom to make it not stuck in low obstacles and for platform jump assistance
2) set FixedRotation to true
3) optionally disable impulses in contact callbacks, so the only separation part is applied (haven't tried it myself, may not work)
4) foot sensor to check if character is grounded
5) other sensors of needed - maybe to grab objects...
6) If you need your character not to interact with certain object - read about fixture filter mask, categories and groups in box2d documentation

When I was making a box2d platformer, all the issues were solved by making main character a rolling hedgehog (like Sonic) =)

Re: Box2D, sensor overlaps

by Vavius » Wed Oct 16, 2013 5:35 pm

Honestly, I'll shrink those 4 options to only 2 options:
1) Use box2d for everything
2) Do not use box2d at all

Choose second only if you are making a mario-like tiled platfromer where the only collision are between AABB rectangles. If you need some inclined surfaces, polygon collisions, real physics on some environment objects (ropes, chains, bridges) - then just use box2d.

Bridging together two physics engines is a real pain. I've tried to make a SPH water to respect dynamic and static box2d bodies, and it a mess (static are working OK, but dynamic aren't), requires C++ coding...
