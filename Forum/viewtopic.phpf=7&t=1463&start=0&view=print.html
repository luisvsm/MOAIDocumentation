I've been using LuaJIT for a while, although JIT is disabled on iOS, it still makes quite a difference on old device (I tested on iPod3).
Moai includes a precompiled Lua script (src/lua-headers/moai.lua), because JIT's bytecode format is different from normal Lua, you need to remove precompiled version and run that script in text form.

I tested on android and iOS, no problem found so far. It's a seamless replacement.
Interpreter LuaJIT is about 2x faster than plain Lua on iOS. I believe it reduces some overhead of function/coroutine calling. But it's still not fast enough to take heavy job.

BTW, you should take care of Lua gc setting, default gc setting is not aggressive enough, so your game might take more peak memory and the collection cycle will be more noticeable (makes your game laggy).

Re: Moai + Luajit

Posted: Wed Jan 30, 2013 8:35 am
by keelo
Ah I figured it out, on XCode at least.

1) remove the reference to the 3rd party/liblua folder from libmoai.xcodeproj
2) add path/to/your/luajit-2.0/src to "Header Search Paths" for libmoai.xcodeproj
3) build LuaJIT with these instructions: http://blog.csdn.net/dragoncheng/articl ... ls/8155079 (I had to remove armv6 from those instructions, because I'm building with iOS SDK 6.1 and I'm unsure how to build armv6 from the command-line with that yet...I think I may need to get an older version of the SDK??)
4) add the resulting luajit-2.0/build/libluajit2.a to your top level xcode project
5) add luajit-2.0/build/libluajit32.a to your top level Mac project, if you're building that, too
6) modify MOAILuaRuntime.cpp to #include lua.h instead of lstate.h, and remove the code that depends on regular lua internals, like in spacepluk's branch above.

Build everything in xcode.

When you run your project, you'll get errors about missing MOAILayer2D, etc. now. This is because the precompiled bytecode MOAI header included in moai-dev/src/lua-headers/moai_lua.h is for regular lua, and luajit has a different bytecode format. So you have to regen it:

7) run "cd moai-dev/src/lua-headers && moai main.lua" with your new moai binaries to generate a new version of the moai.lua bytecode in moai_lua.h
8) build again

It should work!

Hopefully this saves someone else some time.

9) make your damn game

Re: Moai + Luajit

Posted: Sun Mar 03, 2013 9:35 am
by tommo.zhou
It's because MOAI replaced std file functions with zlcore, you have to include that when compile LuaJIT.
This is my build script:
Code: Select all
  1.  

  2. #!/bin/sh

  3. MOAIROOT=/Users/tommo/dev/moai-dev

  4. NDK=/Users/tommo/dev/android-ndk-r8

  5. NDKABI=8

  6. NDKVER=$NDK/toolchains/arm-linux-androideabi-4.4.3

  7. NDKP=$NDKVER/prebuilt/darwin-x86/bin/arm-linux-androideabi-

  8. NDKF="--sysroot $NDK/platforms/android-$NDKABI/arch-arm"

  9. CFLAGS="-I $MOAIROOT/src -include zlcore/zl_replace.h "

  10. rm -rf android

  11. mkdir android

  12.  

  13. make HOST_CC="gcc -m32" CROSS=$NDKP TARGET_CFLAGS="$CFLAGS" TARGET_FLAGS="$NDKF" TARGET=arm TARGET_SYS=android clean all

  14. cp -p src/libluajit.a android/libluajit.a

  15. make clean

  16. cp src/luaconf.h src/lua.h src/lauxlib.h src/lualib.h android

  17.

Tommo, you're a genius. Thank you.

For anyone else who wants to use Tommo's script -- it will appear to error out because running make in luajit's directory will first make the static and shared libraries, and then try to make the luajit executable. The luajit executable build process will explode with linker errors. However the libraries have already been built and the script will pick them up and put them in the right place.

Also, using tommo's branch directly didn't work for me; for some reason my (fairly recent) android sdk errors out when attempting to import android.widget.EditText as part of the MOAIAndroidKeyboard work. Since I don't need MOAIAndroidKeyboard, rather than screw with it I grafted tommo's ant directory onto spacepluk's less recently updated moai-dev. The frankenstein repo then shambled slowly to life.
