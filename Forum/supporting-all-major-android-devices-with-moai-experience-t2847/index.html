Supporting all major Android devices with Moai - Experience

by sshukul » Tue Oct 07, 2014 10:28 am

This is more of a "sharing experience" and possible pull request thread than a question, so maybe other devs can benefit from it.

Basically I realised that for the longest time my Android release for had an unusually high refund rate of around 20% as a paid game on Google Play, and many periodic "black screen" or "extremely slow" 1 star reviews on the store. The game in question is this one here:

https://play.google.com/store/apps/deta ... games.moai

So the first thing I realised was that it is essential to build the Android native .so binaries for all 3 major instruction sets - armeabi, armeabi-v7, and x86. Ignoring any of these three would guarantee a significant portion of the market being broken and can make the difference between a successful and unsuccessful game due to bad reviews. These can be set or passed to the android.toolchain.cmake file at the time of the android host build. The only instruction set that can safely be ignored is mips ... I have had a mips version sitting on Google play for months now without a single download, so it's safe to say no major Android devices use this instruction set.

Once built the binary .so files can either all be copied into the libs folder of the Android host, which will increase it's size significantly but make sure that the device will choose the correct one at runtime, or build separate hosts for each chipset if size is an issue, and use Google Play's multiple version feature to decide at download time which one the device gets.

The other thing which I didn't notice anyone on here mention before, is an issue with the NEON instruction set within armeabi-v7, that cannot simply be accounted for with separate binaries.

The problem is that some armeabi-v7 devices do not support the NEON instruction set, but the Android NDK does not auto-detect and deal with this situation, so in addition to the basic binaries above, in order to eliminate black screens entirely, one must also do an "armeabi-v7 without neon" binary build and rename the generated so to something else like libmoai-no-neon.so and copy it into the libs/armeabi-v7 folder of the host. Then, one must manually add run-time detection code to account for this situation; I added this into the MainActivity.java file of the Android host as below:

Code: Select all
  1.  

  2. //----------------------------------------------------------------//

  3.         static {

  4.                

  5.                 MoaiLog.i ( "Loading libmoai.so" );

  6.                

  7.                 if((Build.CPU_ABI != null && Build.CPU_ABI.equals("x86")) ||

  8.                                 (Build.CPU_ABI2 != null && Build.CPU_ABI2.equals("x86"))) {

  9.                         MoaiLog.i ( "loading moai lib" );

  10.                         System.loadLibrary("moai");

  11.                 } else if((Build.CPU_ABI != null && Build.CPU_ABI.equals("armeabi-v7a")) ||

  12.                                 (Build.CPU_ABI2 != null && Build.CPU_ABI2.equals("armeabi-v7a"))) {

  13.                         boolean hasNeon = hasNeon();

  14.                         if(hasNeon) {

  15.                                 MoaiLog.i ( "Has neon, loading default neon lib" );

  16.                                 System.loadLibrary("moai");

  17.                         } else {

  18.                                 MoaiLog.i ( "No neon, loading non neon lib" );

  19.                                 System.loadLibrary ( "moai-no-neon" );

  20.                         }

  21.                 } else {

  22.                         MoaiLog.i ( "loading moai lib" );

  23.                         System.loadLibrary("moai");

  24.                 }

  25.         }

  26.  

  27.         private static boolean hasNeon() {

  28.                 String archInfo = getArchInfo();

  29.                 MoaiLog.i ( archInfo );

  30.                 return archInfo.toLowerCase(Locale.ENGLISH).contains("neon");

  31.         }

  32.        

  33.         private static String getArchInfo() {

  34.             StringBuffer sb = new StringBuffer();

  35.             if (new File("/proc/cpuinfo").exists()) {

  36.                 try {

  37.                     BufferedReader br = new BufferedReader(new FileReader(new File("/proc/cpuinfo")));

  38.                     String aLine;

  39.                     while ((aLine = br.readLine()) != null) {

  40.                         sb.append(aLine + "\n");

  41.                     }

  42.                     if (br != null) {

  43.                         br.close();

  44.                     }

  45.                 } catch (IOException e) {

  46.                     e.printStackTrace();

  47.                 }

  48.             }

  49.             return sb.toString();

  50.         }

  51.  



Only after all these binaries and detections were added did I finally have an Android host with zero black screen complaints and very few refunds (< 5%).

Don't know if we want to make a pull request out of this or just something people can refer to, but I thought having a stable Android host that supports all major devices is probably fairly useful. If we want do a pull request, I guess I could change the standard make-host.sh script to generate for all 4 instruction set combinations by default, and also make the neon code above to fall back to default in case it doesn't find the one it's looking for.

Code: Select all
  1.  

  2.  if( NEON )

  3.   set( CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -mfpu=neon" )

  4.   set( CMAKE_C_FLAGS "${CMAKE_C_FLAGS} -mfpu=neon" )

  5.  



NEON should only defined if you set the target to "armeabi-v7a with NEON" in the cmake scripts.

Code: Select all
  1.  

  2. if( ARM_TARGET STREQUAL "armeabi" )

  3.  set( ARMEABI true )

  4.  set( ARMEABI_NDK_NAME "armeabi" )

  5.  set( NEON false )

  6. else()

  7.  if( ARM_TARGET STREQUAL "armeabi-v7a with NEON" )

  8.   set( NEON true )

  9.   set( VFPV3 true )

  10.  elseif( ARM_TARGET STREQUAL "armeabi-v7a with VFPV3" )

  11.   set( VFPV3 true )

  12.  endif()

  13.  set( ARMEABI_V7A true )

  14.  set( ARMEABI_NDK_NAME "armeabi-v7a" )

  15. endif()

  16.  



It doesn't explicity set it to false, but it also doesn't set it to true either (it is not a cache variable so shouldn't persist between runs). But if it is crashing in the wild, then thanks for the feedback! I will have to look harder at where the neon code is getting invoked from.

by sshukul » Wed Oct 08, 2014 10:02 am

@halfnelson I did have an x86 build sitting out on the Google Play store from long before the time I received the complaint, and versioned higher than the arm builds (using Google Play's multiple version feature), so in theory if the client device had an x86 chip, it would have gotten just the x86 build.

The reason I did it with multiple versions instead of fitting all the binaries into one host apk was firstly to keep the download size < 50Mb and not have to bother with setting up expansion files, and secondly to be able to easily track the rough distribution of the instruction sets without adding more detailed analytics.

In theory that means not having the x86 binary probably wouldn't have been the reason for the crash on the galaxy tab 3, as Google should have had the device specs and given them the x86 version APK if that was the case, but it could be a case of Google having the wrong device specs and not the moai host being the problem. Seems unlikely though, I would imagine Google Play gets the specs directly from the Android OS.

I also know for a fact that directly sending the armeabi-v7 with neon build apk to the customer with this Galaxy Tab 3 directly fixed the issue, as he emailed back and confirmed that he was able to get the game to run. This could possibly be a case of the tablet running the arm version running slower through emulation on an x86 chip though, but if that's the it's still odd that the original x86 version download didn't work for him if the Galaxy Tab actually had an x86 chipset, as the x86 version ran fine for hundreds of others around that time.

I remember looking up the device specs at the time and found that the problem chipset was the Marvell PXA869, as listed here:

http://www.gsmarena.com/samsung_galaxy_ ... 0-5422.php

I think I remember dealing with another email and finding another Marvell chipset as the culprit too. I think Samsung only put them in a handful of tablet models and the rest have standard Qualcomm, NVidia and Exynos chipsets which seem to run both with and without neon versions fine, but it's these handful odd ones that occasionally caused a bad review / refund and the run-time detection seems to have eliminated that.
