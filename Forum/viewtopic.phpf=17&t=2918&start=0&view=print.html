by ibisum
Based on this and your other report regarding multiple bugs on Android, I'm getting a 'feeling' that maybe there are some lock contention/context-switch issues still going on with the Android host - but what makes this difficult to assess is not knowing which version of the Android host you're using, and so on. There are very easy ways to reproduce crashing bugs on Android, and one of them involves trying to do non-UI things on the UI thread, callbacks in native API context, and the fact that it all starts with a GL context and all that entails on the buggy wasteland that is Android, in general, when things don't quite fit together properly.

So, with this and the other threads you've started on the Android bugs/host-hacks issue (cf., Multi-touch Android hack), it'd be good to know what branch you're building, any of your own local modifications, and so on ..

Re: Android resume and graphic context corruption

by petercummins89
Basically the trick is to detach your GLSurfaceView from the view hierarchy from your Activity's onPause(). Since it's not in the view hierarchy at the point onPause() runs, the context never gets destroyed.

So your Activity's onPause() should look like this:

@Override
public void onPause() {
view.setVisibility(View.GONE);
super.onPause();
...
}

And you restore your GLSurfaceView to the hierarchy not from onResume() but from onWindowFocusChanged() :

@Override
public void onWindowFocusChanged(boolean hasFocus) {
super.onWindowFocusChanged(hasFocus);
if (hasFocus && view.getVisibility() == View.GONE) {
view.setVisibility(View.VISIBLE);
}
...
}

Note that you never call the GLSurfaceView's onPause() and onResume() and that this is the official SDK GLSurfaceView, no hacked-up alternative version is required.
