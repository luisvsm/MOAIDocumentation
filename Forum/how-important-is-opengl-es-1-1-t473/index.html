Re: How important is OpenGL ES 1.1?

by patrick » Wed Feb 08, 2012 10:21 pm

It gets nasty once we start exposing a more complete feature set. There's a of fixed function cruft to write objects for and cache. Good example is the multi-texture stuff we added recently. It's easy enough to model the texture units and cache them, but supporting things like texture combiners and coordinate generation means a lot of extra state to manage. Then on top of that we need to re-implement all of the functionality using shaders if we want to be forward compatible with 2.0. None of it is hard... just more code to write and maintain.

That said, there *are* still a lot of 1.1 devices out there, so point taken.

Anyone else have feelings about this?

Re: How important is OpenGL ES 1.1?

by thebullno1 » Wed Feb 15, 2012 3:31 pm

I think backward compatibility is enough. I have never used stuffs like texture combiner or complex blend mode in the fixed function pipeline. Frankly, I find writing shaders more straightforward than having to figure out what does those flags do (GL_ONE, GL_ZERO...)

How about a MOAIGfxDeviceEx API? It will have OpenGL ES 2.0 support. At runtime, if OpenGL ES 2.0 is detected, that object will be registered, if not it is nil and developers need to handle it somehow. Same goes for functions that requires OpenGL ES 2.0, just nullify them or replace them with a dummy one that prints some warning to the log.

The fixed functions that I'd like to see emulated are:
- Immediate mode drawing (MOAIDraw), they should be easy to emulate using vertex array.
- Some blend mode like additive, multiplicative, subtractive... shouldn't be too hard.
- Bitmap masking would be nice
Anything fancier than that should be 2.0 only features.
