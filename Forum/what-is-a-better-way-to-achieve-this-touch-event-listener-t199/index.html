by bslayerw » Tue Nov 15, 2011 4:21 pm

you should probably be using MOAIPartition, create an instance. Insert all 2200 targets into that and then instead of testing 2200 times you call partition.propListForPoint or parition.propListForRect. This will return a list of objects at x,y. This is way faster because A contains a spatial data structure that organizes the props to make things like this efficient.

by patrick » Wed Nov 16, 2011 1:59 am

That's right. The data structure used by MOAIPartition is a multi-level grid. By default it's a grid of only one level and cell, so acts like a list (but should still be faster than running the loop in Lua). If you know something about the distribution of objects in your game, you can set the partition's dimensions, number of levels and cell sizes. You can set up the partition to emulate a quad tree, a loose quad tree or a custom layout of your own choosing. The partition wraps at the edges, so you don't have to worry about objects going off one edge and onto another.

Ideally, in a real game scenario your 2200 objects will be of roughly regular sizes and evenly distributed throughout the world. In that case you should be able to still get good performance if you set the grid cell size to be a few times larger than the average object size and set the dimensions to be large enough to cover most of your game world (or at least a big chunk of it).

I should also mention that the partition is used for viewport culling too.
