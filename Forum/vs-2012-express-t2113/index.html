Re: VS 2012 Express

by halfnelson » Sun Oct 27, 2013 12:50 pm

Here you go:
https://s3.amazonaws.com/moaicli.moaiforge.com/moaicli_windows_jre.7z

requires windows and comes bundled with its own embedded jre (so you don't need to install one that adds plugins to browsers!)

install:
unzip (7zip) somewhere (program files?) and add the extracted location to your path (or add a batch file to windows\system that points to the batch file in the extracted program)

usage:

create a project with:
Code: Select all
  1. moaicli new myproject


this will create a folder myproject in the current directory

then in the myproject folder
Code: Select all
  1. moaicli start windows


since this is the first time it will generate a config file that you can modify if you wish and tell you about it.

run the command again and it will

  • checkout an up to date branch of the moai-sdk (includes all fixes/mods from dev branch) from the moaiforge repository (needed due to cmake mods) (under appdata\local\moaicli\sdks)
  • download and install cmake (under appdata\local\moaicli\deps\cmake)
  • download and install mingw64 (under appdata\local\moaicli\deps\mingw64)
  • compile moai.exe and place it under myproject\build\windows\windows\app with everything under myproject\src
  • run the resulting moai.exe from that folder


It should be smart enough to not rebuild the entire moai.exe every time you run the start command. It will just copy the lua files on later runs.

to build the same project for android it is just
Code: Select all
  1. moaicli build android


this downloads required sdk,ndk,jdk,cmake,platform-tools and builds an apk under build\android\android. The downloads are large (not my fault, these are from Google. Google chose a rubbish compressor for their ndk distribution)

Code: Select all
  1. moaicli start android


runs the code on a plugged in android device.

Would love to know if this worked fine for you (works for me) and any feedback to make it smoother. I am currently just finishing off iphone launching on device (launching on simulator etc works) before I do a final polish and release the final linux mac and windows binaries.

Re: VS 2012 Express

by halfnelson » Fri Nov 01, 2013 10:47 pm

glad it worked for you.

Batch File: The batch file is a legacy thing before I decided that adding the cli to the path was the best option. If you have the moaicli folder on the path, then removing the batch file from the project directory completely should fix it.

What is moaiforge: Consider it plan B. When the moai devs were awol and looked not interested in updates to the moai-sdk, I setup a git repo/organisation for an unofficial community fork, so that even if zipline didn't come to the party, the community could soldier on (the beauty of opensource).

Zipline eventually woke up and gave the community access to their repository. I decided that moaiforge should stick around anyway to provide a stable default branch for moaicli stuff and was eventually planning to host plugins for moaicli instead of having them in the core sdk (there are the beginnings of plugin support in moaicli). I also needed a domain name to launch/preview my moaijs host, and moaiforge.com wasn't taken.

All of my future moai contributions will be under this namespace and git repo. Moaicli is just the first step in making moai easier for newcomers, I plan on hosting improved tutorials, a copy of the Api documentation that can be commented on (ala php.net) and added to, and an in browser editing/running environment for creating/playing with snippets/examples of moai (ala jsfiddle.net) to assist with the improved documentation and tutorials.

I am happy to plug away on this stuff alone, but wouldn't mind help either, so the moaiforge organisation in github allows me to add contributors with commit access, so that I don't have to manually merge everything.

cmake-development branch: The cli relies on cmake to do its building (so I don't need to update vstudio project files, xcode projects and linux make files for each change to moai-sdk). The cmake mods have not yet been pulled into the main moai-dev branch. So the cmake-development branch on moaiforge is the default branch for new projects (it is up to date with latest moai-dev/development and tested on all platforms).

If you have your own branch/clone/fork of moai and you want to use the cli, just merge in any changes found in the cmake-development branch to get the build environment up to date and modify your project config (config/config.yml) to point to your repository and branch, when you next build it will checkout that version and build against that. If the cmake mods get accepted into mainline, then i will update the project template to point the default sdk location to there.
